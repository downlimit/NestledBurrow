# Library: карта NestledBurrow

Этот файл — карта города. Он помогает найти нужную область проекта, не загружая в контекст всё подряд.

## Как пользоваться картой

- Основной чат сначала читает `PROJECT.md`, чтобы понять текущий вектор, затем `REVIEW.md` перед проверкой pull request.
- Codex или субагент сначала читает `AGENTS.md`, затем эту карту и только после этого открывает адреса, необходимые для задачи.
- Карта указывает маршрут, но не пересказывает содержимое документов.

## Действующие здания

### `PROJECT.md` — правительство

Текущее положение проекта, цель, следующий шаг, блокеры и решения, недавно изменившие направление.

Основной читатель: главный чат, управляющий работой.

Читать: перед планированием следующей задачи или после долгого перерыва.

### `REVIEW.md` — контрольная служба

Канонический протокол основного чата для поиска, классификации, проверки, исправления, merge и публикации pull request. Фиксирует три класса review, пакетное внесение исправлений, дисциплину повторных CI, обязательный просмотр визуальных артефактов, asset-approval gate и правило ожидания `pages/live`.

Основной читатель: главный чат, проверяющий работу Codex.

Читать: перед каждой проверкой PR и при изменении CI/review-процесса.

## Технический район

### `AGENTS.md`

Обязательные правила работы Codex: подготовка, автоматические проверки, ручной runtime self-test, эффективная передача на review, протокол подключения сторонних spritesheet и формат отчёта о проверке.

Читать: в начале каждой задачи Codex.

### `ASSETS.md`

Канонический адрес внешних ассетов, их источников, лицензий, выбранных файлов и итоговых путей в проекте.

Читать: при добавлении, удалении или проверке сторонней графики и лицензий.

### `src/`

Исходный код игры.

Читать: при реализации или проверке игрового поведения.

### `src/main.js`

Точка входа и Phaser-сцена непрерывного мира: загрузка semantic atlas, рендер indoor/outdoor, игрок, камера, анимации, клавиатура, джойстик и экранный HUD.

Читать: при изменении запуска, сцены, камеры, движения игрока, рендера мира или конфигурации Phaser.

### `src/input.js`

Каноническая логика ввода: параметры мобильного джойстика, dead zone, ограничение длины вектора и определение touch/coarse pointer устройств.

Читать: при изменении клавиатурного или мобильного управления.

### `src/visualConfig.js`

Runtime-конфигурация кадров персонажа, foot box, facing hysteresis и скорости анимации.

Читать: при изменении персонажа, анимаций или визуального поведения игрока.

### `src/kenneyRoomConfig.json`

Канонический машинный манифест Kenney-ассетов и утверждённых preview: происхождение pack, геометрия исходного spritesheet, source frame ID и SHA-256, semantic frame names, wall bands, atlas hashes, параметры мира и pixel hashes preview.

Читать: при аудите, замене или повторном выборе тайлов. Raw frame ID не должны переноситься из этого файла в gameplay-код.

### `src/worldConfig.js`

Канонические размеры логического экрана, мира, тайла, размещения комнаты, дверного проёма, скорости игрока и путей semantic atlas непрерывного мира.

### `src/worldLayout.js`

Чистая сборка непрерывного мира: outdoor fill, встроенная комната, dirt path, дверной проём и набор заблокированных клеток.

### `src/movement.js`

Чистая логика foot-box collision, world bounds и axis-separated movement без Phaser Physics.

### `src/roomLayout.js`

Чистая логика сборки прямоугольной комнаты из смысловых имён `floor`, боковых стен, wall bands и дверного проёма.

Читать: при изменении формы комнаты, количества рядов стен или правил раскладки тайлов.

### `scripts/audit-spritesheet.py`

Переиспользуемый инструмент первичного аудита spritesheet: проверяет геометрию frame/margin/spacing, нарезает все кадры, создаёт CSV с координатами и SHA-256 и подписанные contact sheets.

Читать и запускать до выбора любых тайлов из нового большого spritesheet. Он не выбирает ассеты автоматически — смысловой выбор делается только после просмотра contact sheets и, при неоднозначности, подтверждения пользователя.

### `scripts/check-input.mjs`

Автоматические проверки расчётов и ключевых сценариев ввода.

Читать: при изменении логики джойстика, объединения направлений или ограничений движения.

### `scripts/check-visual.mjs`

Автоматические проверки единой pixel grid, файлов персонажа, направлений и idle, целостности room/world semantic atlas, SHA-256 и структуры визуальной конфигурации.

Читать: при изменении визуальных ассетов, манифеста, масштаба или конфигурации комнаты.

### `scripts/check-world.mjs`

Автоматические проверки размеров мира, doorway, wall collision, маршрута indoor→outdoor, sliding и world bounds.

Читать: при изменении мира, двери, collision или движения.

### `scripts/check-room-preview.py`

Генератор и проверка утверждённых preview-артефактов непрерывного мира: полного overview, двух camera view 320×180 и увеличенного фрагмента верхних стыков. Сравнивает их пиксели с утверждёнными SHA-256 из `src/kenneyRoomConfig.json`.

Читать: при изменении atlas, раскладки, масштаба или визуальной регрессии мира.

### `requirements-dev.txt`

Закреплённая Python-зависимость Pillow для воспроизводимого аудита spritesheet и генерации предпросмотров в локальной среде и GitHub Actions.

Читать: при изменении Python-инструментов визуального результата.

### `package.json`

Зависимости и команды запуска, сборки, предпросмотра и технических проверок. Команда `npm run check` объединяет input, visual, world, preview и production build.

Читать: при настройке окружения, зависимостей, сборки или команд проверки.

## Инфраструктурный район

### `.github/workflows/pr-check.yml`

Текущая проверка pull request: последовательная установка Node/Python-зависимостей, все project checks, production build и загрузка `world-previews-pr-<номер>` с четырьмя визуальными артефактами.

Читать: при изменении обязательных проверок PR, распараллеливании fast/visual jobs, path-фильтрации или workflow-артефактов.

### `.github/workflows/deploy-pages.yml`

Сборка и публикация GitHub Pages, генерация `/version.json`, проверка живой опубликованной версии и выставление статуса `pages/live`.

Читать: при изменении публикации, идентификатора сборки или правила «можно запускать».

## Архив

Старые handoff-файлы, Telegram-экспорт и прежние документы памяти не являются действующими адресами. Обращаться к ним нужно только для конкретного вопроса или восстановления утраченного обоснования.

## Правила развития города

- Новое здание появляется только после возникновения самостоятельной области знаний.
- Для здания заранее определяются назначение, читатель и момент чтения.
- Один факт имеет один канонический адрес; другие документы только ссылаются на него.
- Нельзя создавать пустые здания на будущее или дублировать сведения.
- При появлении, переименовании или удалении здания обновляется эта карта.
