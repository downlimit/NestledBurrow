# Binary import bridge

## Назначение

Этот документ фиксирует подтверждённый путь доставки пользовательских бинарных attachments в `downlimit/NestledBurrow` без ручной работы пользователя и без передачи доставки Codex.

## Распределение ответственности

Пользователь только прикладывает файл к Lead-чату и описывает требуемую интеграцию.

Лид обязан:

1. прочитать байты реального attachment;
2. определить канонический repository path;
3. импортировать именно эти байты в GitHub;
4. считать готовый файл обратно и проверить формат, размер и SHA-256;
5. довести импорт до `main` либо включить готовый бинарник в явно указанный base commit;
6. выдать Codex одну code-only задачу с точным Base SHA и путём.

Пользователь не загружает файл в GitHub вручную, не создаёт asset branch и не передаёт тот же attachment Codex.

## Источник истины

Источник истины — байты приложенного пользователем файла.

Внешняя ссылка, CDN URL, исходная страница генератора или файл с тем же названием **не заменяют attachment**. Их разрешено использовать как транспорт только после проверки, что размер и SHA-256 полностью совпадают с attachment.

При первом несовпадении размера или SHA-256 Лид останавливает этот транспортный путь. Запрещено:

- менять ожидаемый hash под скачанный URL;
- импортировать «похожую» версию композиции;
- создавать новые временные workflow для повторных догадок;
- открывать Codex-задачу с отсутствующим runtime-файлом.

## Два технических пути

### Небольшой бинарник

Когда attachment вместе с base64 encoding помещается в допустимый размер аргумента GitHub-коннектора, Лид использует обычную цепочку Git Data API:

```text
attachment bytes
→ create_blob(base64)
→ create_tree
→ create_commit
→ update_ref
→ PR / merge
```

### Крупный бинарник

Когда base64-представление не помещается в один аргумент коннектора, Лид использует repository-side reconstruction bridge.

1. Создать ветку только с префиксом:

```text
asset-import/<slug>
```

2. Разбить base64 attachment на небольшие UTF-8 chunks и подготовить их локально.

3. Создать manifest:

```json
{
  "version": 1,
  "targetPath": "public/assets/audio/music/example.mp3",
  "byteLength": 3977087,
  "sha256": "lowercase-64-character-sha256",
  "chunks": ["0000.b64", "0001.b64"]
}
```

4. Одним Git tree/commit добавить **все** chunks и manifest в:

```text
.binary-import/<import-id>/
```

Промежуточные staging-коммиты не должны запускать Actions.

5. После проверки полного дерева отдельным последним commit добавить пустой marker:

```text
.binary-import/<import-id>/READY
```

Только `READY` запускает `.github/workflows/binary-import.yml`.

6. Workflow один раз вызывает `scripts/reconstruct-binary-import.mjs`, собирает настоящий бинарник, проверяет размер и SHA-256, удаляет всю staging directory и коммитит готовый файл в ту же ветку.

7. Лид проверяет final branch read-back. Asset PR открывается или переводится в final state только после появления настоящего runtime-файла.

В конечном PR и `main` должны находиться настоящий `.mp3`, `.png`, `.ttf` и другие runtime-файлы. Временные `.b64`, manifest и `READY` не должны оставаться в final diff.

## Антиспам-инварианты

Один импорт пользовательского файла должен создавать не более одного reconstruction run.

Запрещено:

- запускать workflow на каждый chunk, manifest edit или диагностический commit;
- открывать PR до завершения staging, если открытие само запускает проверки и review;
- добавлять asset-specific jobs в общий `pr-check.yml`;
- добавлять временный workflow в `main` ради одного файла;
- многократно менять поле `trigger` только ради повторного запуска;
- загружать диагностические копии файла как artifacts, если они не являются необходимым единственным путём восстановления.

Если single-shot run завершился ошибкой, Лид сначала читает полный лог и делает не более одного консолидированного repair. При неопределённости импорт останавливается без новых push.

## Ограничения безопасности

Bridge:

- разрешает target только внутри `public/assets/`;
- запрещает path traversal и ненормализованные пути;
- принимает только разрешённые binary extensions;
- выполняет strict base64 round-trip validation;
- проверяет заявленный byte length и SHA-256 до записи файла;
- сохраняет staging при ошибке для диагностики;
- удаляет staging только после успешной реконструкции;
- не запускается для произвольных веток, только для `asset-import/**`;
- запускается только при добавлении `READY`.

## Codex handoff

Codex не отвечает за импорт пользовательского attachment.

Его prompt после завершённого импорта обязан содержать:

- точный Base SHA;
- точный repository path;
- runtime URL/path, когда отличается;
- SHA-256 и применимые метаданные;
- явный факт, что бинарник уже импортирован и проверен;
- запрет скачивать, регенерировать, переименовывать или повторно загружать файл.

Если файл отсутствует в указанном base, Codex останавливается до реализации. Лид не выдаёт такой prompt заранее.
