# NestledBurrow: bootstrap проекта

## Назначение

`PROJECT.md` — первая точка входа для любого нового ChatGPT-чата внутри Project NestledBurrow. Он хранит минимальный общий контекст, текущее опубликованное состояние и маршрутизацию по ролям.

Детальные операционные правила не дублируются здесь:

- `LEAD.md` — постоянный чат **Лид** для обсуждения, архитектуры и постановки задач Codex;
- `REVIEW.md` — постоянный чат **Интегратор** для поиска, проверки, исправления, merge и публикации PR;
- `AGENTS.md` — обязательный контракт Codex-исполнителя;
- `LIBRARY.md` — необязательная карта файлов и систем;
- `ASSETS.md` — правила внешних ассетов.

Codex не читает `PROJECT.md`, `LEAD.md` или `REVIEW.md` по умолчанию. Для обычной реализации он получает самодостаточный прямой промпт, читает `AGENTS.md` и только релевантные исходники.

## Восстановление нового чата

Внутри ChatGPT Project пользователю достаточно первой фразы:

```text
привет, ты лид
```

или:

```text
привет, ты интегратор
```

Новый чат обязан до первого содержательного ответа:

1. прочитать `PROJECT.md`;
2. зафиксировать роль на весь чат;
3. для роли Лид прочитать `LEAD.md`;
4. для роли Интегратор прочитать `REVIEW.md`;
5. открыть только необходимые дополнительные документы и исходники;
6. не просить пользователя пересказывать проект, перечислять Markdown-файлы или объяснять GitHub-процесс.

Если роль явно не названа:

- обсуждение игры, архитектуры и подготовка задач относятся к Лиду;
- PR, ветки, CI, merge и публикация относятся к Интегратору.

Вне ChatGPT Project остаётся аварийный вход:

```text
В репозитории downlimit/NestledBurrow прочитай PROJECT.md и выбери роль по моему запросу.
```

## Разделение ролей

- **Пользователь** формулирует продуктовый и художественный вижн и оценивает результат в игре.
- **Лид** всегда доступен для разговора, разбирает архитектурные последствия, создаёт параллельно безопасные Codex-промпты и сохраняет устойчивые решения.
- **Интегратор** независимо и последовательно обслуживает GitHub: находит все готовые PR, строит порядок, проверяет, исправляет, сливает и публикует.
- **Codex** реализует одну конкретную техническую задачу в одной ветке по прямому промпту и `AGENTS.md`.

Полное ревью не должно блокировать чат Лида. Пользователь продолжает обсуждать игру в Lead-чате, пока отдельный Integrator-чат занимается PR.

## Операционные команды

Для Интегратора фраза:

```text
проверь все PR
```

означает полный проход по **всем открытым PR в `downlimit/NestledBurrow`, направленным в `main`**. Пользователь не обязан называть партию, номера PR, ветки, SHA или порядок merge.

Фразы `сделал PR`, `проверь PR`, `прими PR`, `исправь и смержи`, `проверь публикацию`, `посмотри ветку` также являются командами к исполнению, а не просьбами дать план.

Интегратор следует `REVIEW.md`, самостоятельно обнаруживает GitHub-инструменты, получает diff, CI и артефакты, исправляет существующие ветки и продолжает до фактического результата или реального технического блокера.

## Получение GitHub-инструментов

Если нужная функция не загружена в текущий контекст:

1. использовать прямой namespace `GitHub`, когда функция уже доступна;
2. иначе вызвать `api_tool.list_resources` для пути `GitHub` с коротким запросом по действию, например `fetch_pr_patch`, `workflow`, `artifact`, `merge`, `branch` или `update_file`;
3. вызвать обнаруженную функцию и продолжить.

Наличие описания функции без загруженной схемы не является ограничением среды. Блокер разрешено заявить только после фактической попытки обнаружения и вызова с указанием конкретной ошибки.

## Где находится проект

Репозиторий: `downlimit/NestledBurrow`.

Опубликованная игра: https://downlimit.github.io/NestledBurrow/.

Технологии: Phaser 4.2.1, Vite, браузерная 2D-игра с логическим разрешением 320×180.

## Текущее опубликованное состояние

- Единый непрерывный top-down мир на CC0-тайлах Basic Village.
- Игрок и два патрулирующих NPC используют одну композиционную `Character`-сущность с общими движением, collision foot box, facing, анимацией и depth sorting.
- Player и patrol controllers возвращают нормализованный `ControllerCommand` с `moveDirection`, `aimDirection` и actions; `Character` передаёт контроллеру изолированный snapshot своего состояния.
- Домашний NPC патрулирует замкнутый loop-маршрут внутри дома; уличный NPC ходит по ping-pong-маршруту вдоль дорожки.
- NPC имеют ту же максимальную скорость, что и игрок, но вдвое меньшие acceleration, braking, reverse acceleration и turn deceleration.
- Все три персонажа временно используют совместимые CC0-ассеты Kenney.
- Velocity-based движение с разгоном, торможением, разворотом и collision foot box.
- Клавиатурное управление и динамический мобильный джойстик объединяются в единый нормализованный movement vector.
- `MobileJoystick` владеет Phaser/native/window pointer lifecycle, capture/fallback, safety reset, графикой и cleanup; активный pointer продолжает управлять после выхода за canvas и потери capture.
- `MovementDebugPanel` владеет опциональной DOM-панелью tuning, persistence, reset/copy, status и cleanup; `WorldScene` остаётся composition root.
- Камера следует только за игроком с целочисленным pixel-grid рендером.
- В Phaser HUD находятся пиксельный идентификатор сборки и кнопка fullscreen.
- GitHub Pages публикует только проверенный `main`; статус `pages/live` подтверждает фактически доступный SHA.

## Текущая цель

Развивать прототип короткими проверяемыми итерациями, сохраняя Lead-чат свободным для живого обсуждения и перенося медленную GitHub-интеграцию в отдельный постоянный Integrator-чат.

## Текущий блокер

Инфраструктурных блокеров нет.

## Общий рабочий цикл

```text
Пользователь обсуждает игру с Лидом
→ Лид выдаёт один или несколько параллельно безопасных Codex-промптов
→ Codex создаёт финальные PR с integration metadata
→ пользователь пишет Интегратору «проверь все PR»
→ Интегратор самостоятельно строит dependency order, проверяет, исправляет и сливает все готовые PR
→ pages/live: success
→ пользователь проверяет опубликованную игру
```

Пользователь не возвращается в промежуточные чаты, не сообщает каждый номер PR и не управляет служебными Batch/Task ID.

## Устойчивые решения

- Основное проектирование и обсуждение происходят в Lead-чате; лимиты Codex преимущественно расходуются на код.
- Полное ревью, merge и публикация происходят в отдельном Integrator-чате.
- Обычная задача использует одну временную ветку и один финальный PR.
- Параллельные задачи получают внутренние Batch/Task metadata, зависимости, merge phase и области владения от Лида.
- `main` защищён от удаления и force-push; GitHub автоматически удаляет merged head-ветки.
- PR открывается только после завершения реализации и применимых проверок.
- Объём процесса и доказательств пропорционален риску изменения.
- Нельзя добавлять зависимости, обходные реализации, тестовые послабления или инфраструктуру только из-за локальной ошибки сети, прокси или установки пакета.
- Визуальное или интерактивное изменение не сливается только по успешной сборке: требуется осмотр фактического результата или честно зафиксированное ограничение с последующей пользовательской проверкой.
- Basic Village Tileset от Forchild — канонический набор окружения; новые внешние ассеты проходят правила `ASSETS.md`.
- Сторонние spritesheet-кадры нельзя выбирать по догадке: значения централизуются и подтверждаются preview.

## Сохранение и актуальность контекста

Репозиторий является источником контекста, который способен восстановить новый чат. Мысль, существовавшая только в незавершённом разговоре и не ставшая подтверждённым решением, восстановить гарантированно нельзя.

После подтверждения устойчивого решения Лид обязан без напоминания пользователя обновить канонического владельца факта. Интегратор перед merge проверяет документальный drift и после публикации сверяет `PROJECT.md` и `LIBRARY.md` с фактическим `main`.

- опубликованное состояние, роли и устойчивые решения → `PROJECT.md`;
- правила Лида и параллельной постановки → `LEAD.md`;
- правила Интегратора, merge и публикации → `REVIEW.md`;
- обязательные правила Codex → `AGENTS.md`;
- важные адреса проекта → `LIBRARY.md`;
- внешние ассеты → `ASSETS.md`.

Нельзя записывать будущую или непроверенную реализацию как уже существующую.
