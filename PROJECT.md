# NestledBurrow: текущий вектор

## Где находится проект

Создан минимальный браузерный прототип на Phaser 4.2.1 и Vite. Он показывает первую пиксельную комнату на проверенных CC0-тайлах Kenney, маленького анимированного персонажа из совместимого набора Kenney, движение с клавиатуры и мобильный виртуальный джойстик. В правом верхнем углу отображается идентификатор опубликованной сборки.

Репозиторий: `downlimit/NestledBurrow`.

Опубликованная игра: https://downlimit.github.io/NestledBurrow/.

## Текущая цель

Инфраструктурный рабочий путь настроен. Текущая цель — развивать сам игровой прототип короткими проверяемыми итерациями, не добавляя инфраструктурные усложнения без фактической необходимости.

## Следующий шаг

Следующая игровая задача определяется в основном чате. Заранее придумывать её и фиксировать без обсуждения не нужно.

## Текущий блокер

Текущих инфраструктурных блокеров нет.

## Рабочий цикл

`ChatGPT формулирует ограниченную задачу → Codex делает изменения и выполняет обязательный self-test из AGENTS.md → пользователь нажимает «Создать PR» → ChatGPT проверяет код, CI и фактический визуальный результат → при необходимости исправляет или очищает ветку → выполняет merge → ждёт pages/live: success → сообщает «можно запускать»`

## Действующие решения

- Основное общение, проектирование и постановка задач происходят в обычном ChatGPT-чате.
- Лимиты Codex расходуются преимущественно на непосредственную работу с кодом.
- Игра создаётся как браузерная 2D-игра на Phaser 4.
- Код и проектные документы хранятся в GitHub.
- Изменения проходят через отдельные ветки и pull request.
- Корневой `AGENTS.md` задаёт обязательный порядок self-test для Codex: автоматические проверки, запуск игры и проверка фактических изменённых состояний.
- Сторонний spritesheet нельзя подключать по угаданным числовым индексам. Сначала создаются подписанные contact sheets, проверяются метаданные и примеры автора, затем выбранные кадры переносятся в компактный semantic atlas.
- Gameplay-код обращается к внешним тайлам по смысловым именам. Исходные номера кадров и SHA-256 хранятся только в аудиторском манифесте `src/kenneyRoomConfig.json`.
- Визуальное изменение нельзя сливать только на основании успешной сборки и проверки числовых конфигураций: до merge нужно осмотреть фактический рендер и загруженный workflow-артефакт предпросмотра.
- Pull request должен пройти workflow `Pull Request Check`, который запускает проверки ввода, ассетов, semantic layout, pixel-exact room preview и production-сборку.
- Изменение atlas или layout обязано сохранить утверждённый pixel hash комнаты либо предоставить новый `artifacts/room-preview.png` для осознанного визуального утверждения.
- После merge workflow публикации создаёт `/version.json` с полным и коротким SHA сборки.
- Workflow публикации самостоятельно проверяет живой `/version.json` и выставляет статус `pages/live`.
- Изменение считается опубликованным только после `pages/live: success`.
- Формулировка «можно запускать» используется только после успешной проверки живой версии.
- Для обхода браузерного кэша ссылка на игру может дополняться параметром `?v=<короткий SHA>`.
- Комната использует проверенные CC0-тайлы из Kenney Roguelike/RPG Pack, собранные в небольшой atlas с именами `floor`, `wallOuter`, `wallMiddle`, `wallInner` и вариантами краёв.
- Персонаж использует CC0-ассеты Kenney, отдельный нейтральный idle-кадр и корректные направления ходьбы влево и вправо.
- Следующий игровой шаг определяется в основном чате.
- Реализованы клавиатурное управление и мобильный виртуальный джойстик с обработкой отмены и потери активного касания.
- Проектные знания разделяются на тематические «здания» только по фактической необходимости; список зданий заранее не придумывается.
