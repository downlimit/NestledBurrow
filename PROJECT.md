# NestledBurrow: текущий вектор

## Где находится проект

Создан минимальный браузерный прототип на Phaser 4.2.1 и Vite. Он показывает единый непрерывный top-down мир на CC0-тайлах Basic Village, маленького анимированного персонажа Kenney, velocity-based движение с клавиатуры и динамический мобильный джойстик. В правом верхнем углу находятся идентификатор опубликованной сборки и кнопка полноэкранного режима.

Репозиторий: `downlimit/NestledBurrow`.

Опубликованная игра: https://downlimit.github.io/NestledBurrow/.

## Текущая цель

Развивать игровой прототип короткими проверяемыми итерациями, не добавляя инфраструктурные усложнения без фактической необходимости.

## Следующий шаг

Следующая игровая задача определяется в основном чате. Заранее придумывать её и фиксировать без обсуждения не нужно.

## Текущий блокер

Текущих инфраструктурных блокеров нет.

## Рабочий цикл

`ChatGPT формулирует ограниченную задачу по tasks/TEMPLATE.md → Codex использует одну временную ветку от актуального main без открытого PR → выполняет обязательный self-test из AGENTS.md → PR создаётся один раз после завершения реализации и локальных проверок → ChatGPT проверяет полный релевантный diff, CI и фактический визуальный/runtime-результат → при необходимости пакетно исправляет ту же ветку → выполняет merge → ждёт pages/live: success → проверяет автоматическое удаление временной ветки → сообщает «можно запускать»`

## Действующие решения

- Основное общение, проектирование и постановка задач происходят в обычном ChatGPT-чате.
- Лимиты Codex расходуются преимущественно на непосредственную работу с кодом.
- Игра создаётся как браузерная 2D-игра на Phaser 4.
- Код и проектные документы хранятся в GitHub.
- Обычные изменения проходят через одну временную ветку и один финальный pull request.
- `main` защищён от случайного удаления и force-push на стороне GitHub.
- GitHub автоматически удаляет временную head-ветку после merge; оставшиеся старые ветки очищаются только отдельной безопасной задачей `tasks/branch-cleanup.md`.
- Черновой PR не открывается во время разработки: он создаётся один раз после прохождения локальных проверок, чтобы не запускать промежуточные CI и не создавать лишние уведомления.
- Корневой `AGENTS.md` задаёт обязательный порядок работы Codex: одна ветка, автоматические проверки, runtime-проверка, контроль зависимостей, запрет несвязанных обходных решений и формат отчёта.
- `REVIEW.md` задаёт канонический порядок проверки PR, пакетных исправлений, merge, публикации и проверки удаления временной ветки.
- `tasks/TEMPLATE.md` задаёт обязательный Git lifecycle для новых задач.
- `.github/pull_request_template.md` задаёт стандартный доказательный отчёт по финальному head commit.
- Локальная ошибка сети, прокси или установки зависимости не является основанием добавлять самодельную замену пакета, ослаблять тесты или менять CI вне задачи.
- **Basic Village Tileset от Forchild является приоритетным и каноническим набором окружения.** Другие наборы используются только для отсутствующих категорий или после отдельного визуального согласования.
- Персонаж пока использует совместимые CC0-ассеты Kenney.
- Сторонний spritesheet нельзя подключать по угаданным числовым индексам. Сначала проверяются геометрия листа, подписанные кадры и примеры автора; выбранные индексы централизуются в конфигурации и подтверждаются фактическим preview.
- Визуальное или интерактивное изменение нельзя сливать только на основании успешной сборки и проверки числовых конфигураций: до merge нужно осмотреть фактический рендер, workflow-артефакт и доступные runtime-состояния.
- Pull request должен пройти workflow `Pull Request Check`, который запускает проверки ввода, fullscreen, движения, ассетов, layout, preview и production-сборку.
- После merge workflow публикации создаёт `/version.json` с полным и коротким SHA сборки.
- Workflow публикации самостоятельно проверяет живой `/version.json` и выставляет статус `pages/live`.
- Изменение считается опубликованным только после `pages/live: success`.
- Формулировка «можно запускать» используется только после успешной проверки живой версии.
- Для обхода браузерного кэша ссылка на игру может дополняться параметром `?v=<короткий SHA>`.
- Реализованы клавиатурное управление и динамический мобильный джойстик: он появляется под первым допустимым касанием в левой половине экрана, сохраняет фиксированный центр до отпускания и сбрасывается при отмене, blur, потере касания и fullscreen-переходах.
- Реализована DOM-кнопка fullscreen для контейнера игры с синхронизацией состояния и integer zoom.
- Проектные знания разделяются на тематические «здания» только по фактической необходимости; список зданий заранее не придумывается.

## Текущая игровая сцена

Прототип использует единый непрерывный top-down мир в одной Phaser-сцене. Интерьер дома собран из Basic Village и встроен в окружающую травяную область; игрок проходит через двухтайловый дверной проём без перехода сцен, камера следует за игроком в пределах мира, а весь world art и персонаж отрисовываются на единой 16×16 logical pixel grid при разрешении 320×180 с integer zoom.
