# NestledBurrow: контекст виртуального лида

## Аудитория

`PROJECT.md` предназначен для основного ChatGPT-чата, который выполняет роль виртуального лида проекта.

Codex **не читает этот файл по умолчанию**. Для обычной реализации Codex получает самодостаточный прямой промпт, читает `AGENTS.md` и только релевантные исходники. `PROJECT.md` передаётся Codex только когда конкретная задача действительно зависит от продуктовой истории или стратегического решения, которое нельзя достаточно точно включить в прямой промпт.

## Восстановление нового основного чата

Это единственная обязательная точка входа для восстановления контекста проекта. Пользователю достаточно отправить:

```text
В репозитории downlimit/NestledBurrow прочитай PROJECT.md.
```

После чтения основной чат обязан самостоятельно:

1. восстановить текущее состояние, роли и рабочий цикл;
2. понять **текущий запрос пользователя как команду к исполнению**, а не только как тему для обсуждения;
3. открыть только необходимые подробные документы и файлы;
4. проверить актуальное состояние PR, веток или публикации в GitHub, когда запрос относится к ним;
5. не заставлять пользователя пересказывать историю проекта или обслуживать внутренний процесс.

## Операционный запуск нового чата

Фразы вроде `сделал PR`, `проверь PR`, `прими PR`, `исправь и смержи`, `проверь публикацию` или `посмотри ветку` являются операционными командами. После восстановления контекста основной чат обязан в том же ходе перейти к GitHub-инструментам и выполнить применимую часть `REVIEW.md`.

Нельзя заканчивать ответ планом, пересказом инструкции, списком будущих шагов или просьбой прислать diff/номер PR, когда репозиторий известен и данные можно найти самостоятельно.

Если нужная GitHub-функция не загружена в текущий контекст:

1. использовать прямой namespace `GitHub`, когда функция уже доступна;
2. иначе вызвать `api_tool.list_resources` для пути `GitHub` с коротким запросом по нужному действию, например `fetch_pr_patch`, `workflow`, `merge` или `branch`;
3. вызвать обнаруженную функцию и продолжить задачу.

Наличие описания функции без уже загруженной схемы **не является ограничением среды**. Заявлять о недоступности инструмента можно только после фактической попытки обнаружения и вызова. В блокере должны быть названы конкретная неудавшаяся операция и полученная ошибка, а не предположение модели о своих возможностях.

Вопрос пользователя во время активного ревью не останавливает работу: основной чат кратко отвечает и продолжает вызовы инструментов в том же ходе.

## Разделение ролей

- **Пользователь** формулирует продуктовый и художественный вижн и оценивает результат в игре.
- **Основной ChatGPT-чат** владеет декомпозицией, архитектурой, постановкой задач, Git, ревью, исправлениями, merge, публикацией и актуальностью канонических документов.
- **Codex** является исполнителем конкретной технической задачи в пределах прямого промпта и `AGENTS.md`.

Пользователь не обязан отслеживать противоречия в Markdown-файлах, напоминать об их обновлении, выбирать стратегию веток или вручную обслуживать GitHub.

## Маршрутизация документов

- `PROJECT.md` — лидовский контекст, текущее состояние, устойчивые продуктовые и процессные решения.
- `REVIEW.md` — протокол основного ChatGPT-чата для проверки, исправления, merge и публикации PR. Codex не читает его по умолчанию.
- `AGENTS.md` — единственный обязательный репозиторный документ для Codex перед обычной реализацией.
- `LIBRARY.md` — необязательная карта адресов. Открывается только когда неизвестно, где находится нужная система или документ.
- `ASSETS.md` — источники и правила внешних ассетов; читается только в задачах, затрагивающих ассеты.
- `tasks/TEMPLATE.md` и конкретные `tasks/*.md` — только для явно выбранных крупных, рискованных, многоэтапных или возобновляемых задач.

Стандартное начало обычного ключа для Codex:

```text
Прочитай AGENTS.md и только релевантные файлы проекта.
```

Не следует автоматически добавлять в ключ `PROJECT.md`, `REVIEW.md` или `LIBRARY.md`.

## Где находится проект

Репозиторий: `downlimit/NestledBurrow`.

Опубликованная игра: https://downlimit.github.io/NestledBurrow/.

Технологии: Phaser 4.2.1, Vite, браузерная 2D-игра с логическим разрешением 320×180.

## Текущее опубликованное состояние

- Единый непрерывный top-down мир на CC0-тайлах Basic Village.
- Игрок и два патрулирующих NPC используют одну композиционную `Character`-сущность с общими движением, collision foot box, facing, анимацией и depth sorting.
- Домашний NPC патрулирует замкнутый loop-маршрут внутри дома; уличный NPC ходит по ping-pong-маршруту вдоль дорожки.
- NPC имеют ту же максимальную скорость, что и игрок, но вдвое меньшие acceleration, braking, reverse acceleration и turn deceleration.
- Все три персонажа временно используют совместимые CC0-ассеты Kenney.
- Velocity-based движение с разгоном, торможением, разворотом и collision foot box.
- Клавиатурное управление и динамический мобильный джойстик.
- Активный native pointer продолжает управлять джойстиком после выхода за canvas и потери pointer capture через глобальный `window` fallback; matching `pointerup`/`pointercancel` и safety-события корректно сбрасывают ввод.
- Камера следует только за игроком с целочисленным pixel-grid рендером.
- В Phaser HUD находятся пиксельный идентификатор сборки и кнопка fullscreen.
- GitHub Pages публикует только проверенный `main`; статус `pages/live` подтверждает фактически доступный SHA.

## Текущая цель

Развивать игровой прототип короткими проверяемыми итерациями: от пользовательского вижна до опубликованного игрового результата без лишней подготовительной бюрократии и без накопления скрытого технического долга.

## Текущий блокер

Инфраструктурных блокеров нет.

## Рабочий цикл

`Пользователь описывает вижн → основной чат анализирует продуктовые и архитектурные последствия → выдаёт один самодостаточный ключ Codex → Codex работает в одной временной ветке и создаёт один финальный PR после self-test → основной чат проверяет diff, CI и применимый runtime/visual результат → пакетно исправляет ту же ветку при необходимости → merge → pages/live: success → проверка удаления временной ветки → пользователь получает ссылку на готовую игру и оценивает результат`.

Отдельный task-файл или подготовительный PR не являются стандартным этапом.

## Устойчивые решения

- Основное проектирование и постановка задач происходят в обычном ChatGPT-чате; лимиты Codex преимущественно расходуются на код.
- Обычная итерация использует одну временную ветку и один финальный PR.
- `main` защищён от удаления и force-push; GitHub автоматически удаляет merged head-ветки.
- PR открывается только после завершения реализации и применимых проверок.
- Объём процесса и доказательств пропорционален реальному риску изменения.
- Нельзя добавлять зависимости, обходные реализации, тестовые послабления или инфраструктуру только из-за локальной ошибки сети, прокси или установки пакета.
- Визуальное или интерактивное изменение не сливается только по успешной сборке: требуется осмотр фактического результата или честно зафиксированное ограничение с последующей пользовательской проверкой.
- Basic Village Tileset от Forchild — канонический набор окружения; новые внешние ассеты проходят правила `ASSETS.md`.
- Сторонние spritesheet-кадры нельзя выбирать по догадке: значения централизуются и подтверждаются preview.

## Владелец актуальности документации

Основной ChatGPT-чат обязан поддерживать канонические документы без дополнительных напоминаний пользователя.

Перед merge каждого PR основной чат проверяет документальный drift:

- изменилось опубликованное состояние, продуктовая архитектура или рабочий цикл → обновить `PROJECT.md`;
- добавились, удалились или переместились важные адреса проекта → обновить `LIBRARY.md`;
- изменились обязательные правила исполнителя → обновить `AGENTS.md`;
- изменился процесс ревью и доставки → обновить `REVIEW.md`;
- изменились источники или правила ассетов → обновить `ASSETS.md`.

Правки выполняются в том же implementation PR во время ревью, когда это безопасно. Отдельный documentation PR используется только когда сама документация является самостоятельным изменением.

Нельзя записывать в канонические документы будущую или непроверенную реализацию как уже существующую. После merge и публикации основной чат ещё раз сверяет фактический результат с `PROJECT.md` и `LIBRARY.md`.
