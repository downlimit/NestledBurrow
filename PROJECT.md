# NestledBurrow: bootstrap проекта

## Назначение

`PROJECT.md` — первая точка входа для нового ChatGPT-чата внутри Project NestledBurrow. Он хранит минимальный общий контекст, текущее опубликованное состояние, устойчивые решения и маршрутизацию по ролям.

Детальные правила и знания находятся в отдельных владельцах:

- `GAME.md` — канонический продуктовый вижн, игровой опыт, подтверждённые решения и открытые гипотезы;
- GitHub issue `#80` (`Canonical execution ledger`) — что уже поставлено, что выполняется, что завершено, что принято или отклонено пользователем и какой slice идёт следующим;
- `LEAD.md` — постоянный чат **Лид**: обсуждение игры, архитектура и постановка задач Codex;
- `REVIEW.md` — постоянный чат **Интегратор**: поиск, проверка, исправление, merge и публикация PR;
- `AGENTS.md` — обязательный контракт Codex-исполнителя;
- `LIBRARY.md` — карта важных файлов и систем;
- `ASSETS.md` — правила внешних ассетов.

Codex не читает `PROJECT.md`, `GAME.md`, `LEAD.md` или `REVIEW.md` по умолчанию. Для обычной реализации он получает прямой самодостаточный промпт, читает `AGENTS.md` и только релевантные исходники. Номер задачи из issue `#80` передаётся в prompt и PR body как ссылка на восстановимый execution context, но Codex не обязан читать весь журнал.

## Восстановление нового чата

Пользователю достаточно первой фразы:

```text
привет, ты лид
```

или:

```text
привет, ты интегратор
```

Новый чат обязан до первого содержательного ответа:

1. прочитать `PROJECT.md`;
2. зафиксировать роль на весь чат;
3. для роли Лид прочитать `LEAD.md`, открыть issue `#80`, проверить актуальный `main`, открытые PR и последние релевантные merges; `GAME.md` открывать, когда вопрос касается продукта или roadmap;
4. для роли Интегратор прочитать `REVIEW.md`, открыть issue `#80`, затем самостоятельно найти все применимые открытые PR;
5. открыть только необходимые дополнительные документы и исходники;
6. не просить пользователя пересказывать проект, перечислять Markdown-файлы, вспоминать выданные задачи или объяснять GitHub-процесс.

Если роль явно не названа:

- обсуждение игры, архитектуры и подготовка задач относятся к Лиду;
- PR, ветки, CI, merge и публикация относятся к Интегратору.

## Разделение ролей

- **Пользователь** формулирует продуктовый и художественный вижн, владеет балансом и тюнингом и оценивает результат в игре.
- **Лид** остаётся доступным для живого обсуждения, проверяет актуальный `main`, выделяет минимальный playable slice, принимает архитектурные решения, обновляет execution ledger и выдаёт готовые Codex-задачи.
- **Интегратор** самостоятельно обслуживает GitHub: находит готовые PR, выбирает достаточную глубину проверки, исправляет, сливает и публикует, после чего обновляет completion record соответствующей задачи в issue `#80`.
- **Codex** реализует одну конкретную техническую задачу в одной ветке по прямому промпту и `AGENTS.md`.

Полное ревью не должно блокировать Lead-чат. Пользователь продолжает обсуждать игру с Лидом, пока отдельный Integrator-чат занимается PR.

## Операционные команды

Для Интегратора фраза:

```text
проверь все PR
```

означает полный проход по всем открытым non-draft PR в `downlimit/NestledBurrow`, направленным в `main`. Пользователь не обязан называть партию, номера PR, ветки, SHA, зависимости или порядок merge.

Фразы `сделал PR`, `проверь PR`, `прими PR`, `исправь и смержи`, `проверь публикацию` и `посмотри ветку` также являются командами к исполнению, а не просьбами дать план.

## Где находится проект

Репозиторий: `downlimit/NestledBurrow`.

Опубликованная игра: https://downlimit.github.io/NestledBurrow/.

Канонический execution ledger: https://github.com/downlimit/NestledBurrow/issues/80.

Технологии: Phaser 4.2.1, Vite, браузерная 2D-игра с логическим разрешением 320×180.

## Текущее опубликованное состояние

- Единый непрерывный top-down мир на CC0-тайлах Basic Village.
- Игрок и два патрулирующих NPC используют общий `Character`-агрегат из независимого от Phaser `CharacterMotor` и Phaser-представления `CharacterVisual`.
- `CharacterSystem` хранит персонажей по стабильным ID, централизованно обновляет их и выдаёт runtime-free snapshots.
- Неизменяемые actor profiles `player` и `villager` являются каноническими источниками movement-параметров; runtime-конфигурации создаются отдельными mutable-копиями.
- Отдельный immutable registry visual profiles независимо назначает внешний вид: player использует исходные cardinal Kenney PNG и утверждённый diagonal spritesheet, home NPC — зелёно-бирюзовые cardinal/diagonal sheets, street NPC — сине-бордовые cardinal/diagonal sheets. Все три имеют отдельные animation prefixes и общий нормализованный frame-reference contract.
- Чистый восьмисекторный quantizer выбирает cardinal или diagonal facing по плавно поворачивающемуся motor direction, удерживает направление у границы через hysteresis и сохраняет последнее направление после остановки.
- Walk cycle использует `step A → neutral → step B → neutral`; после остановки любой персонаж явно возвращается в neutral frame текущего из восьми направлений.
- Player и patrol controllers возвращают нормализованный `ControllerCommand` с movement, aim и actions.
- NPC двигаются примерно втрое медленнее игрока. Северный `home-npc` использует замкнутый многоточечный `loop`; южный `street-npc` проходит творческий многоточечный `ping-pong` вперёд и по тому же списку назад. Оба маршрута содержат pass-through точки без остановки и смысловые точки отдыха по 2–3 секунды.
- Collision resolver получает bounds, cell size и blocking query через явный environment contract.
- `GameSessionState` хранит JSON-сериализуемые player/NPC entities, durable flags и transient dialogue state без Phaser/runtime references.
- Версионированный save envelope сохраняет только устойчивый прогресс, безопасно восстанавливается после повреждённого/неподдерживаемого save и не смешивает язык с игровым состоянием.
- Первый законченный мини-квест связывает домашнего и уличного NPC: поручение, ответ, возвращение и repeat-реплики определяются стабильными dialogue/flag IDs.
- Interaction targets строятся из свежих `CharacterSystem` snapshots; во время разговора блокируется movement input игрока и приостанавливается только выбранный NPC.
- EN/RU локализация построена на i18next, ICU namespaces и Pixelify Sans; чистое хранилище стартует на русском, а выбранный язык сохраняется отдельно от игрового состояния.
- `GameHud` владеет build label, fullscreen, language toggle, локализованным `NEW GAME` и sound panel; сброс удаляет игровой progress, но сохраняет language/audio preferences.
- Версионированные audio settings хранят master/music/effects, а runtime проигрывает зацикленную `NestledBurrow_SunlitSavePoint.mp3` с безопасным fallback при недоступном storage.
- `InteractionHud` рисует Unicode prompt/dialogue panel и отделяет touch interaction от мобильного джойстика.
- Клавиатура и динамический мобильный джойстик объединяются в единый нормализованный movement vector; desktop/mobile Chromium E2E покрывает язык, quest flow, reload persistence, reset, audio preferences и live-проверку диагонального facing с сохранением направления после остановки.
- `MobileJoystick` и `MovementDebugPanel` сохраняют самостоятельное lifecycle ownership; `WorldScene` остаётся composition root.
- Камера следует за игроком с целочисленным pixel-grid рендером; GitHub Pages публикует только проверенный `main`, а `pages/live` подтверждает фактически доступный SHA.

## Текущая цель

Максимально ускорять получение играбельного результата и продуктового знания, сохраняя зрелые архитектурные границы там, где они уже доказали пользу.

Каноническое направление игры определено в `GAME.md`: игрок превращает заброшенный летающий остров и таверну в живое персональное заведение, добывая ресурсы, настраивая процессы, обслуживая гостей и постепенно формируя собственное окружение.

Фактический статус активной задачи и последовательность следующих slices находятся в issue `#80`. На момент этой записи активна `NB-GAMEPLAY-001 — Clear the first debris object` со статусом `Ready for Codex`.

Ближайший playable slice:

1. один засоряющий участок объект можно разобрать через существующий interaction flow;
2. действие тратит настраиваемую энергию и выдаёт ресурс;
3. удаление объекта и полученный результат устойчиво переживают save/reload;
4. ключевые значения доступны для быстрого runtime-тюнинга;
5. реализация проверяет продуктовое ощущение расчистки и присвоения пространства, не создавая общий framework ради будущих механик.

Следующие продуктовые ступени после подтверждения этого slice: полный цикл одного растения, первый простой цикл приготовления и продажи напитка, один минимальный миньон и первая реакция аудитории на стиль таверны. Детальный порядок и product acceptance log принадлежат issue `#80`.

Приоритет отдаётся коротким законченным vertical slices, а не рефакторингу ради будущей универсальности и не процессной церемонии ради самой церемонии.

## Основной принцип процесса

**Fast lane используется по умолчанию. Строгий режим включается по реальному риску, а не для каждого PR.**

Качество сохраняется за счёт:

- ясного наблюдаемого результата;
- одного продуктового вопроса для gameplay-эксперимента;
- небольшого scope;
- критических архитектурных инвариантов;
- автоматических проверок;
- targeted review областей, которые CI не способен доказать;
- фактического пользовательского осмотра видимых и интерактивных изменений.

Качество не повышается от повторения одной и той же проверки несколькими слоями или от обязательного дробления маленькой работы на множество PR.

## Рабочие режимы

### Fast lane — обычный режим

Используется для самостоятельных gameplay, UI, content, config, локальных refactor и исправлений с понятной областью.

```text
обсуждение с Лидом
→ запись одобренного slice в issue #80
→ одна компактная самодостаточная Codex-задача
→ один финальный PR со ссылкой на task ID
→ CI + targeted diff review
→ применимый smoke test
→ merge
→ completion record и пользовательский verdict в issue #80
→ итоговый CI/pages на main
```

Для fast lane не обязательны Batch/Task ID, merge phase, owned-path ceremony, отдельный documentation PR или повторный ручной прогон того, что уже надёжно доказал CI. Короткий execution ID из issue `#80` нужен не как церемония, а как восстановимая связь между планом, prompt, PR, merge и пользовательским verdict.

### Strict lane — по риску

Включается, когда изменение затрагивает хотя бы одну из областей:

- центральные архитектурные границы или владение состоянием;
- movement/collision/input contracts с широким regression radius;
- persistence, serialization schema или migrations;
- dependencies, workflows, deployment или security-sensitive конфигурацию;
- общие registries и entry points, изменяемые несколькими параллельными задачами;
- внешние ассеты, лицензии или массовые изменения данных;
- несколько зависимых PR, где неправильный merge order способен повредить результат.

Тогда используются подробные Integration metadata, fan-out/fan-in при реальной необходимости, более строгие targeted checks и документация изменившихся контрактов.

## Ускоряющие устойчивые решения

- Обычная работа оформляется как **одна задача → одна ветка → один PR**.
- Параллелизм используется только когда он реально сокращает календарное время и задачи имеют раздельные области владения.
- Fan-out/fan-in не применяется автоматически; маленький вертикальный срез предпочтительнее трёх служебных PR.
- Одобренная задача до выдачи Codex получает компактную запись в issue `#80`; это не требует branch, commit, PR или CI.
- Codex-промпт связывает execution ID, product question, observable result, relevant anchors, critical invariants, task-specific exclusions, acceptance/evidence и primary review risk. Постоянные правила не копируются из `AGENTS.md` без необходимости.
- Интегратор не повторяет вручную проверки, уже надёжно выполненные CI; review концентрируется на diff, архитектуре, primary risk и рисках, не покрытых автоматикой.
- Независимый PR с непересекающимися файлами не обязан автоматически ребейзиться и полностью перепроверяться после merge другого независимого PR. Совместимость очереди подтверждается итоговым CI на `main`.
- Зависимые, конфликтующие и high-risk PR синхронизируются с новым `main` до merge.
- Ручной runtime smoke test обязателен для изменившегося пользовательского поведения, но не для чистой документации или локальной pure-code логики с достаточными тестами.
- Каноническая документация обновляется только при изменении устойчивого факта, публичного контракта, roadmap или процесса.
- Документация по возможности входит в тот же implementation/integration PR или обновляется один раз в конце значимой волны. Отдельный DOC PR после каждого небольшого merge запрещён как стандартная практика.
- Optional cleanup не задерживает полезный merge и возвращается в backlog только при реальной ценности.
- Нельзя добавлять зависимости, обходные реализации, тестовые послабления или инфраструктуру из-за локальной ошибки сети, прокси или установки пакета.
- Визуальное или интерактивное изменение не сливается только по успешной сборке: нужен фактический осмотр либо честно обозначенный residual risk с последующей пользовательской проверкой.

## Сохранение контекста

Репозиторий и связанные с ним GitHub records являются источником восстановимого контекста. Мысль, задача или результат, существовавшие только в незавершённом разговоре, восстановить гарантированно нельзя.

Одобренная задача считается восстановимой только после записи в issue `#80`. Статусы различаются строго:

- `Ready for Codex` — задача поставлена и готова к запуску;
- `In progress` — Codex начал реализацию;
- `PR open` — существует проверяемый implementation PR;
- `Merged` — код находится в `main`, но пользователь ещё может не принять игровой результат;
- `Accepted`, `Revise` или `Rejected` — зафиксирован продуктовый verdict пользователя;
- `Blocked` — назван конкретный внешний или технический блокер.

После подтверждения устойчивого решения Лид без напоминания обновляет канонического владельца факта. После merge Интегратор или Лид обновляет completion record и ссылку на PR/merge SHA в issue `#80`; после пользовательского теста Лид фиксирует product verdict там же.

- продуктовый вижн, игровой опыт, подтверждённые решения и гипотезы → `GAME.md`;
- опубликованное состояние, роли и текущий roadmap direction → `PROJECT.md`;
- активные задачи, очередь slices, completion records и product acceptance log → issue `#80`;
- правила Лида → `LEAD.md`;
- правила Интегратора → `REVIEW.md`;
- обязательные правила Codex → `AGENTS.md`;
- важные адреса → `LIBRARY.md`;
- внешние ассеты → `ASSETS.md`.

Нельзя записывать будущую или непроверенную реализацию как уже существующую. Открытый PR не является завершением, merge не является пользовательским product acceptance, а принятие пользователем не подменяет доказательство merge в `main`.
