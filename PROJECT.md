# NestledBurrow: единая точка входа

## Назначение этого файла

`PROJECT.md` — единственный файл, который пользователь должен просить прочитать в новом чате для восстановления контекста проекта.

После чтения этого файла чат должен самостоятельно:

1. восстановить текущее состояние проекта, рабочий цикл и принятые решения;
2. понять текущий запрос пользователя;
3. открыть только те подробные документы и файлы, которые нужны для этого запроса;
4. не просить пользователя пересказывать историю проекта, перечислять инструкции или объяснять, какие Markdown-файлы существуют;
5. не читать все Markdown-файлы подряд без необходимости.

Дальнейшая маршрутизация:

- `LIBRARY.md` — карта проекта и адреса нужных областей;
- `REVIEW.md` — обязательный протокол проверки, исправления, merge и публикации pull request;
- `AGENTS.md` — обязательные правила для Codex, реализации технических задач, проверки и доставки;
- `ASSETS.md` — канонические источники и правила использования ассетов;
- `tasks/TEMPLATE.md` — необязательный шаблон для крупных, рискованных, многоэтапных или возобновляемых задач;
- конкретный файл из `tasks/` — требования уже начатой сложной задачи, когда основной чат явно решил, что отдельный task-файл оправдан.

Чат сам выбирает, какие из этих документов открыть. Пользователю достаточно сказать: `Прочитай PROJECT.md`.

Если запрос относится к существующему PR, ветке или незавершённой задаче, чат после чтения инструкций должен самостоятельно проверить их текущее состояние в GitHub, а не полагаться на устаревшее описание из предыдущего разговора.

## Где находится проект

Создан минимальный браузерный прототип на Phaser 4.2.1 и Vite. Он показывает единый непрерывный top-down мир на CC0-тайлах Basic Village, маленького анимированного персонажа Kenney, velocity-based движение с клавиатуры и динамический мобильный джойстик. Игрок и два патрулирующих NPC создаются через одну композиционную Character-сущность; домашний NPC ходит по loop-маршруту внутри дома, уличный — по ping-pong-маршруту вдоль дорожки. В правом верхнем углу игры находятся пиксельный идентификатор опубликованной сборки и внутриигровая кнопка полноэкранного режима.

Репозиторий: `downlimit/NestledBurrow`.

Опубликованная игра: https://downlimit.github.io/NestledBurrow/.

## Текущая цель

Развивать игровой прототип короткими проверяемыми итерациями, не добавляя инфраструктурные усложнения без фактической необходимости.

## Рабочий цикл

`Пользователь формулирует вижн и оценивает игровой результат → ChatGPT самостоятельно разбирает архитектурные последствия и задаёт вопрос только при реальной продуктовой неоднозначности → по умолчанию выдаёт один самодостаточный ключ/промпт для Codex без отдельного task-файла и подготовительного PR → Codex использует одну временную ветку от актуального main без открытого PR → выполняет применимый self-test из AGENTS.md → создаёт один финальный PR после завершения реализации и проверок → ChatGPT проверяет релевантный diff, CI и применимый визуальный/runtime-результат → при необходимости пакетно исправляет ту же ветку → выполняет merge → ждёт pages/live: success → проверяет автоматическое удаление временной ветки → сообщает пользователю только продуктовый результат и существенные ограничения`

## Действующие решения

- Пользователь отвечает за вижн, приоритеты и оценку игрового результата. ChatGPT отвечает за декомпозицию, архитектуру, Git, проверки, чистоту проекта, merge и публикацию; пользователь не должен разбираться во внутреннем устройстве пайплайна или вручную обслуживать ветки.
- Основное общение, проектирование и постановка задач происходят в обычном ChatGPT-чате.
- Лимиты Codex расходуются преимущественно на непосредственную работу с кодом.
- Обычная короткая итерация запускается одним прямым промптом Codex. Отдельный task-файл и отдельный подготовительный PR для передачи спецификации не создаются.
- Task-файл оправдан только для крупной, высокорисковой, многоэтапной, возобновляемой или повторно используемой работы, когда отдельный контракт реально снижает риск потери контекста.
- Если реализации нужна актуализация проектной документации, она выполняется в том же implementation PR. Предварительный documentation PR создаётся только когда сама документация является самостоятельным результатом.
- Игра создаётся как браузерная 2D-игра на Phaser 4.
- Код и проектные документы хранятся в GitHub.
- Обычные изменения проходят через одну временную ветку и один финальный pull request.
- `main` защищён от случайного удаления и force-push на стороне GitHub.
- GitHub автоматически удаляет временную head-ветку после merge; оставшиеся старые ветки очищаются только отдельной безопасной задачей `tasks/branch-cleanup.md`.
- Черновой PR не открывается во время разработки: он создаётся один раз после прохождения применимых проверок, чтобы не запускать промежуточные CI и не создавать лишние уведомления.
- Объём процесса, проверки и отчёта должен быть пропорционален реальному риску изменения. Нельзя создавать отдельные документы, таблицы, чеклисты, тестовые обвязки или инфраструктуру только ради формального заполнения процесса.
- Для `docs` и небольших низкорисковых изменений достаточно краткого отчёта по применимым пунктам. Неприменимые разделы удаляются, а не заполняются длинными `N/A`.
- Новые канонические документы создаются только когда информация действительно будет переиспользоваться; иначе обновляется существующий адрес.
- Корневой `AGENTS.md` задаёт обязательный порядок работы Codex: одна ветка, проверки, runtime-инспекция, контроль зависимостей, запрет несвязанных обходных решений и пропорциональный формат отчёта.
- `REVIEW.md` задаёт канонический порядок проверки PR, пакетных исправлений, merge, публикации и проверки удаления временной ветки.
- `tasks/TEMPLATE.md` задаёт минимальную структуру только для тех задач, которым действительно нужен отдельный долговечный контракт.
- `.github/pull_request_template.md` задаёт адаптивный отчёт по финальному head commit: сохраняются только применимые разделы.
- Локальная ошибка сети, прокси или установки зависимости не является основанием добавлять самодельную замену пакета, ослаблять тесты или менять CI вне задачи.
- **Basic Village Tileset от Forchild является приоритетным и каноническим набором окружения.** Другие наборы используются только для отсутствующих категорий или после отдельного визуального согласования.
- Персонаж пока использует совместимые CC0-ассеты Kenney.
- Сторонний spritesheet нельзя подключать по угаданным числовым индексам. Сначала проверяются геометрия листа, подписанные кадры и примеры автора; выбранные индексы централизуются в конфигурации и подтверждаются фактическим preview.
- Визуальное или интерактивное изменение нельзя сливать только на основании успешной сборки и проверки числовых конфигураций: до merge нужно осмотреть фактический рендер, workflow-артефакт и доступные runtime-состояния.
- Pull request должен пройти workflow `Pull Request Check`, который запускает проверки ввода, fullscreen, HUD, движения, ассетов, layout, preview и production-сборку.
- После merge workflow публикации создаёт `/version.json` с полным и коротким SHA сборки.
- Workflow публикации самостоятельно проверяет живой `/version.json` и выставляет статус `pages/live`.
- Изменение считается опубликованным только после `pages/live: success`.
- Для обхода браузерного кэша ссылка на игру может дополняться параметром `?v=<короткий SHA>`.
- Реализованы клавиатурное управление и динамический мобильный джойстик: он появляется под первым допустимым касанием в левой половине экрана, сохраняет фиксированный центр до отпускания, использует native pointer capture для продолжения drag за пределами canvas и сбрасывается при реальном завершении, отмене, blur, потере capture и fullscreen-переходах.
- Fullscreen и идентификатор сборки находятся внутри Phaser screen-space HUD, используют целочисленное пиксельное выравнивание и не должны захватывать gameplay input.
