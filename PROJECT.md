# NestledBurrow: bootstrap проекта

## Назначение

`PROJECT.md` — точка восстановления долгого продуктового контекста. Для обычной реализации пользователь сразу работает с Codex; отдельная роль нужна только по явной просьбе.

## Канонические владельцы

- `GAME.md` — продуктовый вижн и честная карта фактической реализации: что отсутствует, где есть только фундамент, что играбельно и какой пользовательский verdict зафиксирован.
- `ROADMAP.md` — что делается сейчас, постоянные Task numbers, что идёт следующим и что уже завершено.
- `LEAD.md` — опциональная роль для продуктового разбиения сложной работы.
- `REVIEW.md` — опциональная роль для независимой приёмки нескольких или сложных PR.
- `AGENTS.md` — обязательный контракт Codex.
- `LIBRARY.md` — карта важных файлов и систем.
- `ASSETS.md` — источники, лицензии и хэши внешних ассетов.
- `BINARY_IMPORT.md` — технический контракт импорта пользовательских бинарников.
- `tasks/*.md` — только действительно большие, многоэтапные или возобновляемые технические контракты.

GitHub issues не являются обязательной памятью проекта. Новый чат не должен знать или угадывать номера issues.

## Постоянная идентичность работы

Каждая согласованная задача получает последовательный номер в `ROADMAP.md` до выдачи Codex prompt.

Канонический заголовок:

```text
Task #001 — Первая расчистка участка
```

Он дословно повторяется:

- в первой строке Codex prompt, чтобы автоматически создаваемый заголовок задачи был стандартизирован;
- в PR title;
- в поле `Task:` PR body;
- в сообщениях Лида и Интегратора.

После открытия PR используется Task-first ссылка:

```text
Task #001 — Первая расчистка участка (PR #81)
```

Task number назначается один раз, никогда не переиспользуется и не меняется при repair, rebase или повторной приёмке. PR number является только GitHub-адресом реализации и не заменяет Task number.

## Восстановление нового чата

По умолчанию пользователь ставит задачу непосредственно Codex. Codex сам выполняет branch → Ready PR → CI → merge и не требует отдельного Lead/Integrator handoff.

Пользователю достаточно сказать:

```text
привет, ты лид
```

или:

```text
привет, ты интегратор
```

До первого содержательного ответа:

### Лид

1. прочитать `PROJECT.md`, `LEAD.md`, `GAME.md` и `ROADMAP.md`;
2. проверить актуальный `main`, открытые PR и последние релевантные merges;
3. понять по `GAME.md`, что реально существует и что пользователь уже оценивал;
4. понять по `ROADMAP.md`, какая работа активна, её Task number и что запланировано дальше;
5. не просить пользователя пересказывать игру, сделанные механики или историю задач.

### Интегратор

1. прочитать `PROJECT.md`, `REVIEW.md`, `GAME.md` и `ROADMAP.md`;
2. самостоятельно найти все применимые открытые PR;
3. связать каждый PR с Task number и названием;
4. выполнить команду пользователя, не запрашивая номера PR, ветки, SHA или CI.

Команда `проверь все PR` означает самостоятельно найти и обработать все открытые PR в `main`.

Если роль явно не названа, работает Codex. Он может обсуждать результат, реализовывать, проверять, публиковать и сливать одну задачу end-to-end.

## Роли

- **Пользователь** владеет продуктовым и художественным выбором, балансом и оценкой игрового ощущения.
- **Лид** опционально выделяет playable slice и помогает с продуктовым выбором.
- **Интегратор** опционально проводит независимую приёмку сложных или нескольких PR.
- **Codex** является основным исполнителем: обсуждает задачу, реализует её в одной ветке, открывает Ready PR, ждёт CI, исправляет, сливает и обновляет локальный `main`.

## Репозиторий и runtime

- Репозиторий: `downlimit/NestledBurrow`.
- Опубликованная игра: https://downlimit.github.io/NestledBurrow/.
- Технологии: Phaser 4.2.1, Vite, браузерная 2D-игра, логическое разрешение 320×180.

## Что уже опубликовано

Текущий `main` содержит технический фундамент:

- единый непрерывный top-down мир;
- игрока, камеру, коллизии, keyboard/mobile input и восемь направлений;
- двух патрулирующих NPC;
- диалоги и один устойчивый мини-квест;
- JSON-safe `GameSessionState` и versioned save/reload;
- `NEW GAME`, не сбрасывающий language/audio preferences;
- RU/EN локализацию, HUD, звук и Pages-публикацию;
- runtime movement tuning;
- `WorldScene` как composition root.

Этот список не равен состоянию продукта. Точная зрелость каждой игровой области находится только в `GAME.md`.

## Текущее направление

Каноническая формула игры: игрок превращает заброшенный летающий остров и таверну в живое персональное заведение, добывает и перерабатывает ресурсы, обслуживает гостей, настраивает процессы, создаёт помощников и постепенно собирает собственное поселение из аллодов.

Активный slice, его Task number и следующие проверки находятся в `ROADMAP.md`.

## Основной процесс

```text
короткий запрос пользователя Codex
→ Task # и короткий observable result
→ одна ветка и один финальный PR с тем же Task title
→ targeted local checks один раз
→ Ready PR и полный CI
→ merge и актуальный локальный main
→ пользовательский тест и verdict
→ следующий slice
```

Fast lane используется по умолчанию. Strict lane нужен только для реального риска: persistence, state ownership, movement/input/collision contracts, dependencies, workflows, deployment, security, внешние ассеты или зависимые PR. Детальный аудит скорости и приоритетный checklist находятся в `FAST_LOOP.md`.

## Сохранение контекста

- продукт и состояние механик → `GAME.md`;
- Task numbers, активная и следующая работа → `ROADMAP.md`;
- опубликованный технический bootstrap → `PROJECT.md`;
- правила ролей → `LEAD.md` и `REVIEW.md`;
- правила Codex → `AGENTS.md`;
- кодовые адреса → `LIBRARY.md`;
- ассеты → `ASSETS.md` и `BINARY_IMPORT.md`.

Открытый PR не является завершением. Merge не является пользовательским принятием. Зелёный CI доказывает технический контракт, но не качество игрового ощущения.
