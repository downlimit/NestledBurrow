# NestledBurrow: bootstrap проекта

## Назначение

`PROJECT.md` — первая точка входа для нового ChatGPT-чата. Он объясняет, где лежит восстановимый контекст, какую роль взять и что проверить до содержательного ответа.

## Канонические владельцы

- `GAME.md` — продуктовый вижн и честная карта фактической реализации: что отсутствует, где есть только фундамент, что играбельно и какой пользовательский verdict зафиксирован.
- `ROADMAP.md` — что делается сейчас, что идёт следующим и что уже завершено.
- `LEAD.md` — правила Лида: обсуждение игры, архитектура и подготовка задач Codex.
- `REVIEW.md` — правила Интегратора: PR, CI, repair, merge и публикация.
- `AGENTS.md` — обязательный контракт Codex.
- `LIBRARY.md` — карта важных файлов и систем.
- `ASSETS.md` — источники, лицензии и хэши внешних ассетов.
- `BINARY_IMPORT.md` — технический контракт импорта пользовательских бинарников.
- `tasks/*.md` — только действительно большие, многоэтапные или возобновляемые технические контракты.

GitHub issues не являются обязательной памятью проекта. Новый чат не должен знать или угадывать номера issues.

## Восстановление нового чата

Пользователю достаточно сказать:

```text
привет, ты лид
```

или:

```text
привет, ты интегратор
```

До первого содержательного ответа:

### Лид

1. прочитать `PROJECT.md`, `LEAD.md`, `GAME.md` и `ROADMAP.md`;
2. проверить актуальный `main`, открытые PR и последние релевантные merges;
3. понять по `GAME.md`, что реально существует и что пользователь уже оценивал;
4. понять по `ROADMAP.md`, какая работа активна и что запланировано дальше;
5. не просить пользователя пересказывать игру, сделанные механики или историю задач.

### Интегратор

1. прочитать `PROJECT.md`, `REVIEW.md`, `GAME.md` и `ROADMAP.md`;
2. самостоятельно найти все применимые открытые PR;
3. выполнить команду пользователя, не запрашивая номера PR, ветки, SHA или CI.

Команда `проверь все PR` означает самостоятельно найти и обработать все открытые non-draft PR в `main`.

Если роль явно не названа:

- обсуждение игры, архитектуры и постановка задач → Лид;
- PR, ветки, CI, merge и публикация → Интегратор.

## Роли

- **Пользователь** владеет продуктовым и художественным выбором, балансом и оценкой игрового ощущения.
- **Лид** выделяет минимальный playable slice, принимает архитектурные решения, поддерживает `GAME.md` и `ROADMAP.md`, выдаёт самодостаточные задачи Codex.
- **Интегратор** проверяет и исправляет PR, сливает, подтверждает публикацию и после merge обновляет фактическую зрелость затронутых механик.
- **Codex** реализует одну конкретную техническую задачу в одной ветке по prompt и `AGENTS.md`.

Полная приёмка PR не должна блокировать Lead-чат.

## Репозиторий и runtime

- Репозиторий: `downlimit/NestledBurrow`.
- Опубликованная игра: https://downlimit.github.io/NestledBurrow/.
- Технологии: Phaser 4.2.1, Vite, браузерная 2D-игра, логическое разрешение 320×180.

## Что уже опубликовано

Текущий `main` содержит технический фундамент:

- единый непрерывный top-down мир;
- игрока, камеру, коллизии, keyboard/mobile input и восемь направлений;
- двух патрулирующих NPC;
- диалоги и один устойчивый мини-квест;
- JSON-safe `GameSessionState` и versioned save/reload;
- `NEW GAME`, не сбрасывающий language/audio preferences;
- RU/EN локализацию, HUD, звук и Pages-публикацию;
- runtime movement tuning;
- `WorldScene` как composition root.

Этот список не равен состоянию продукта. Точная зрелость каждой игровой области находится только в `GAME.md`.

## Текущее направление

Каноническая формула игры: игрок превращает заброшенный летающий остров и таверну в живое персональное заведение, добывает и перерабатывает ресурсы, обслуживает гостей, настраивает процессы, создаёт помощников и постепенно собирает собственное поселение из аллодов.

Активный slice и следующие проверки находятся в `ROADMAP.md`.

## Основной процесс

```text
обсуждение с Лидом
→ обновлённые GAME.md и ROADMAP.md
→ одна самодостаточная задача Codex
→ одна ветка и один финальный PR
→ CI + targeted review + применимый runtime smoke
→ merge и публикация
→ обновление зрелости в GAME.md
→ пользовательский тест и verdict
→ следующий slice
```

Fast lane используется по умолчанию. Strict lane нужен только для реального риска: persistence, state ownership, movement/input/collision contracts, dependencies, workflows, deployment, security, внешние ассеты или зависимые PR.

## Сохранение контекста

- продукт и состояние механик → `GAME.md`;
- активная и следующая работа → `ROADMAP.md`;
- опубликованный технический bootstrap → `PROJECT.md`;
- правила ролей → `LEAD.md` и `REVIEW.md`;
- правила Codex → `AGENTS.md`;
- кодовые адреса → `LIBRARY.md`;
- ассеты → `ASSETS.md` и `BINARY_IMPORT.md`.

Открытый PR не является завершением. Merge не является пользовательским принятием. Зелёный CI доказывает технический контракт, но не качество игрового ощущения.
