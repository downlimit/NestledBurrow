# Task #020 — Ускорить цикл Codex

Этот файл фиксирует аудит Task #018 и практический чеклист быстрого цикла. Он нужен людям для оценки процесса; Codex не обязан перечитывать его перед каждой задачей. Исполняемые правила находятся в `AGENTS.md`.

## Что произошло в Task #018

Исходные данные пользователя:

- длительность: около 34 минут;
- расход: больше 20% недельного лимита;
- результат: 21 изменённый файл, 500 добавлений и 83 удаления;
- локальная и GitHub-проверка в итоге зелёные, результат слит.

Точного профилирования по каждой команде не велось, поэтому доли ниже являются качественной оценкой по фактическому журналу действий.

Полезная работа:

- связанная реализация времени, освещения, HUD, audio feedback и движения;
- регрессионные проверки legacy migration и движения;
- один визуально найденный дефект перекрытия HUD;
- финальный CI.

Избыточная работа:

- task prompt на 438 строк и примерно 2 100 слов повторял правила репозитория, проектировал реализацию и перечислял почти каждую проверку;
- targeted checks, `npm run build`, полный `npm run check` и часть runtime/E2E проверяли одно и то же повторно;
- шесть обязательных screenshot состояний дублировали автоматический E2E и ручной runtime smoke;
- локальный платформозависимый PNG hash потребовал отдельной base-проверки, Draft PR, пропущенного CI и повторного перевода PR в Ready;
- `npm ci` и временные worktree использовались из-за заблокированного локального `node_modules`;
- полный diff и длинные успешные логи повторно загружались в контекст;
- GitHub CLI проверялся, хотя установленный GitHub connector уже покрывал PR и CI;
- обязательный отдельный Integrator handoff создавал лишний этап после реализации;
- ценная функция Лида — перевод свободного пользовательского описания в компактное ТЗ с обязательным и достаточно глубоким учётом затронутой архитектуры — была ошибочно смешана с этой бюрократией.

## Приоритетный чеклист

### P0 — влияет на каждую задачу

- [x] Сделать маршрут пользователь → Лид → Codex основным: Лид переводит свободное описание в компактное архитектурно осмысленное ТЗ, Codex реализует его end-to-end.
- [x] Integrator использовать только по прямому запросу или для нескольких зависимых PR.
- [x] Ограничить routine prompt результатом, важными ограничениями и acceptance; не повторять `AGENTS.md`.
- [x] Убрать повторные validation-команды: один локальный путь, один удалённый CI.
- [x] Открывать обычный Ready PR после локальной проверки; Draft не использовать как CI gate.
- [x] После зелёного CI автоматически завершать обычную задачу merge и обновлением локального `main`, если пользователь прямо не запретил merge.
- [x] Называть работу пользователю сначала через `Task #N — название`; PR указывать вторичным адресом в скобках.
- [x] Удалить workflow, ожидающий отдельный Codex review для auto-merge.

### P1 — сильно влияет на большие игровые задачи

- [x] Локально запускать только focused E2E для изменённого поведения; полный E2E оставлять PR CI.
- [x] Для одного material risk использовать одно сильное evidence. Скриншоты нужны только для визуального суждения, обычно не больше двух.
- [x] Не запускать `npm ci`, когда зависимости уже пригодны и dependency/lock contract не менялся.
- [x] Не повторять отдельный `build` после полного `npm run check`, потому что build уже входит в эту цепочку.
- [x] При base failure один раз выполнить точную failing-команду на `main`; не повторять полную suite и не приписывать failure feature diff.
- [x] Использовать GitHub connector первым; GitHub CLI нужен только для отсутствующей операции или логов, которых connector не предоставляет.
- [x] На успехе выводить короткий итог команды; полный лог показывать только при failure.

### P2 — контроль дальнейшей деградации

- [x] Сократить обязательные процессные документы и убрать повторяющиеся правила.
- [x] Сократить PR template до Task, результата, validation и реального limitation.
- [x] Оставить durable `tasks/*.md` только для реально возобновляемой или зависимой работы.
- [ ] Сравнить следующие три задачи по wall time и расходу недельного лимита.
- [ ] Если средняя routine-задача снова выходит за 15 минут до CI, провести новый аудит конкретного bottleneck.

## Целевой быстрый цикл

```text
свободное описание пользователя
→ компактное архитектурно осмысленное ТЗ Лида
→ Task #N и ветка Codex от актуального main
→ минимальная реализация
→ targeted checks один раз
→ Ready PR
→ полный GitHub CI
→ repair в той же ветке только при реальном failure
→ merge
→ локальный main обновлён
→ короткий отчёт пользователю
```

Ориентир для небольшой или средней routine-задачи: 10–15 минут до Ready PR без учёта очереди GitHub CI. Это цель процесса, а не обещание для больших игровых slices.

## Измеримый результат Task #020

- семь основных процессных файлов до оптимизации: 7 606 слов и 1 090 строк;
- после оптимизации: 3 188 слов и 523 строки;
- постоянный процессный контекст сокращён примерно на 58%;
- отдельный review-gated auto-merge workflow удалён;
- PR CI запускается независимо от Draft/Ready статуса;
- routine completion теперь заканчивается merge и обновлением локального `main` без отдельной команды пользователя.

Task #021 уточнила маршрут после обратной связи пользователя: Лид остаётся постоянным переводчиком замысла в ТЗ; сокращение handoff относится к отдельной приёмке Интегратора.

## Что процесс больше не требует по умолчанию

- отдельный Integrator-чат;
- Task-файл для routine iteration;
- Draft PR перед CI;
- ручной Codex review перед merge;
- одновременные unit checks, полный check, полный E2E, ручной smoke и набор screenshots для одного и того же риска;
- полный успешный command log в пользовательском контексте;
- ручные действия пользователя с ветками, PR, CI или merge.
