# Задача: пиксельная комната и анимированный персонаж

## Исходная точка

Работай от актуального `main` репозитория `downlimit/NestledBurrow`.

Перед изменениями прочитай:

- `PROJECT.md`
- `LIBRARY.md`
- `ASSETS.md`, если файл уже существует
- `src/main.js`
- `src/input.js`
- `src/style.css`
- `package.json`

В этой папке должны находиться два временных исходных архива:

- `tasks/pixel-room/source/kenney_rpg-urban-pack.zip`
- `tasks/pixel-room/source/kenney_roguelike-rpg-pack.zip`

Также проверь `tasks/pixel-room/source/SHA256SUMS` и `tasks/pixel-room/source/README.md`.

Если хотя бы одного архива нет, он пустой или не открывается как ZIP:

- ничего не изменяй;
- не создавай коммит;
- напиши только: `PIXEL ROOM SOURCE ARCHIVES ARE MISSING OR INVALID`.

Оба набора — бесплатные CC0-ассеты Kenney. Архивы нужны только как временный источник и не должны остаться в итоговом проекте после выполнения задачи.

## Цель

Заменить технические цветные прямоугольники первой комнаты на простую пиксельную комнату, а прямоугольник игрока — на маленького анимированного персонажа с четырьмя направлениями движения.

Сохранить существующие:

- размер игры `960 × 540`;
- WASD и стрелки;
- мобильный виртуальный джойстик;
- нормализацию диагонального движения;
- ограничение движения стенами;
- build label;
- систему `/version.json` и `pages/live`.

Не добавлять новые игровые механики.

## 1. Изучение архивов и выбор ассетов

Распакуй оба архива во временную директорию вне `public`.

### Персонаж

Из `Kenney RPG Urban Pack` выбери одного обычного человеческого персонажа:

- не транспорт;
- не животное;
- не декоративный объект;
- визуальный размер примерно того же класса, что персонажи небольших top-down RPG;
- должны существовать кадры ходьбы вверх, вниз, влево и вправо.

Перед интеграцией изучи реальные имена файлов, размеры и порядок кадров. Не угадывай раскладку по памяти.

Если в наборе есть единый spritesheet — используй его. Если кадры лежат отдельными PNG — скопируй только выбранного персонажа и только нужные кадры.

### Комната

Из `Kenney Roguelike/RPG Pack` выбери согласованный минимальный набор:

- деревянный или наиболее похожий на деревянный тайл пола;
- верхняя горизонтальная стена;
- нижняя горизонтальная стена;
- левая вертикальная стена;
- правая вертикальная стена;
- четыре угла.

Если стены представлены автотайлом или общим spritesheet, используй минимально необходимый набор областей/кадров. Не копируй весь пакет из сотен или тысяч отдельных файлов в `public`.

## 2. Итоговое размещение ассетов

Добавь только реально используемые файлы:

```text
public/assets/third-party/kenney/player/
public/assets/third-party/kenney/room/
```

Не оставляй в репозитории:

- исходные ZIP-архивы;
- временно распакованные каталоги;
- демонстрационные изображения;
- файлы для других движков;
- неиспользуемые персонажи и тайлы.

В конце задачи полностью удали каталог:

```text
tasks/pixel-room/
```

## 3. Лицензии

Создай или обнови корневой `ASSETS.md`.

Для каждого набора укажи:

- точное название;
- автор: Kenney;
- официальную исходную страницу;
- лицензию: `CC0 1.0 Universal`;
- исходные имена выбранных файлов;
- итоговые пути файлов в проекте;
- какие именно части используются;
- что полные исходные архивы после интеграции удалены.

Официальные страницы:

- `https://kenney.nl/assets/rpg-urban-pack`
- `https://kenney.nl/assets/roguelike-rpg-pack`

Добавь `ASSETS.md` в `LIBRARY.md` как канонический адрес внешних ассетов и лицензий.

## 4. Пиксельный рендер

В конфигурации Phaser включи:

```js
pixelArt: true,
antialias: false,
roundPixels: true,
```

Сохрани `Phaser.Scale.FIT` и `Phaser.Scale.CENTER_BOTH`.

Используй только целочисленный масштаб спрайтов. Базовый ориентир — `ART_SCALE = 3`, но подбери целое значение по реальному размеру выбранных кадров.

Желаемая визуальная высота персонажа: приблизительно `48–64` игровых пикселя.

Не использовать:

- сглаживание;
- дробный scale;
- CSS-фильтры размытия;
- масштабирование отдельных кадров до разных размеров.

## 5. Комната

Удали однотонный фон комнаты и четыре цветных прямоугольника стен.

Собери одну прямоугольную комнату:

1. повторяющийся деревянный пол;
2. верхняя стена;
3. нижняя стена;
4. левая стена;
5. правая стена;
6. четыре корректно соединённых угла.

Выровняй всё по тайловой сетке с учётом выбранного целочисленного масштаба.

Для этой проверки используй простые циклы Phaser. Не добавляй:

- Tiled;
- редактор карт;
- процедурную генерацию;
- несколько комнат;
- мебель;
- дверь;
- освещение;
- тени;
- декор.

Внутренние границы движения рассчитывай из фактических координат и размеров стен, а не из прежней константы `WALL_SIZE`.

## 6. Персонаж и анимации

Замени временный прямоугольник игрока на `Phaser.GameObjects.Sprite`.

Создай состояния:

- `idle-down`;
- `idle-up`;
- `idle-left`;
- `idle-right`;
- `walk-down`;
- `walk-up`;
- `walk-left`;
- `walk-right`.

Если отдельных idle-анимаций нет, используй нейтральный кадр соответствующей walk-анимации. Не создавай искусственное покачивание на месте.

Параметры ходьбы:

- `repeat: -1`;
- ориентир `8 fps`;
- одинаковая частота для всех направлений;
- диагональное движение не должно ускорять анимацию.

Установи origin персонажа по ногам:

```js
setOrigin(0.5, 1)
```

Положение и границы движения рассчитывай относительно нижней центральной точки персонажа, а не центра полного кадра и не прозрачных полей.

Phaser Physics не добавлять.

## 7. Направление движения

Добавь `lastFacing` со значениями:

- `down`;
- `up`;
- `left`;
- `right`.

При движении выбирай направление по преобладающей оси.

Используй гистерезис около `15%`, чтобы при почти диагональном вводе анимация не мигала между горизонталью и вертикалью. Если разница осей находится внутри порога, сохраняй предыдущее направление.

При остановке:

- немедленно останови walk-анимацию;
- покажи idle-кадр текущего `lastFacing`;
- не оставляй случайный промежуточный кадр шага.

Скорость движения сохрани примерно текущей. Уменьшай её только при явной необходимости для соответствия частоте шагов.

## 8. Столкновение со стенами

Персонаж не должен пересекать внутреннюю границу комнаты ногами.

Учитывай:

- ширину области ступней/тела, а не прозрачную ширину кадра;
- нижнюю точку origin;
- фактическую толщину стен;
- возможное визуальное перекрытие головой верхней стены.

Небольшой заход головы на верхнюю стену допустим, если ноги остаются внутри комнаты.

Не добавляй физический движок.

## 9. Слои

Порядок отображения:

1. пол;
2. стены;
3. персонаж;
4. мобильный джойстик;
5. build label.

Джойстик и build label должны остаться поверх игровой графики.

## 10. Организация кода

Не складывай всю новую логику в `create()`.

Допустимо выделить компактные методы, например:

- `preload()`;
- `createRoom()`;
- `createPlayer()`;
- `createPlayerAnimations()`;
- `updatePlayerAnimation()`.

Не создавать:

- ECS;
- систему сущностей;
- крупную архитектуру сцен;
- общий менеджер ассетов;
- новые npm-зависимости.

Обнови `PROJECT.md`:

- технические примитивы комнаты заменены пиксельными тайлами;
- прямоугольник игрока заменён анимированным персонажем;
- используются только CC0-ассеты Kenney;
- следующий игровой шаг определяется в основном чате.

Обнови `LIBRARY.md`:

- добавь `ASSETS.md`;
- уточни роль `src/main.js`, если она изменилась;
- не создавай другие новые Markdown-файлы.

## 12. Проверка

Проверь вручную и кодом, где разумно:

- персонаж виден при запуске;
- пол закрывает внутреннюю область без пустых промежутков;
- стены и углы соединены корректно;
- пиксели не размыты;
- idle работает для четырёх направлений;
- walk работает для четырёх направлений;
- переход `walk → idle` показывает правильный нейтральный кадр;
- направление не мигает при почти диагональном вводе;
- диагональ не увеличивает скорость;
- WASD и стрелки работают;
- мобильный джойстик работает;
- второй палец не перехватывает джойстик;
- отпускание/отмена касания останавливает персонажа;
- персонаж не выходит за стены;
- build label остаётся видимым;
- временная папка `tasks/pixel-room/` удалена;
- в `public` лежат только реально используемые изображения.

Выполни:

```bash
npm ci
npm run check:input
npm run build
```

Сделай один итоговый коммит.

Не выполнять `git push`.
Не использовать `gh`.
После завершения пользователь самостоятельно нажмёт «Создать PR».
