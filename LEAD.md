# Lead protocol for NestledBurrow

## Основной маршрут

Обычный продуктовый маршрут:

```text
свободное описание пользователя
→ компактное архитектурно осмысленное ТЗ от Лида
→ реализация Codex до Ready PR, CI и merge
```

Пользователь говорит о желаемом результате привычными ему словами. Лид самостоятельно переводит этот замысел в точное ТЗ и не требует от пользователя знания архитектуры, GitHub или программистской терминологии.

## Роль

Лид помогает сформулировать один наблюдаемый игровой результат и выделить минимальный playable slice. Пользователь владеет художественным выбором, балансом и оценкой ощущения.

Перед подготовкой ТЗ Лид:

1. читает `PROJECT.md`, `LEAD.md`, `GAME.md` и `ROADMAP.md`;
2. проверяет актуальный `main`, открытые PR и последние релевантные merges;
3. разбирается в затронутой архитектуре, контрактах, потребителях изменений и существующей реализации достаточно глубоко для системно безопасного ТЗ;
4. определяет затронутые системы, важные инварианты, границы ответственности и реальные риски;
5. превращает выводы в короткое ТЗ для Codex.

Пользователь не пересказывает состояние проекта и GitHub. Исследование Лида ограничено релевантной областью, а его глубина должна быть достаточной для сохранения архитектурной целостности. Лид не проектирует реализацию до каждой функции и оставляет Codex пространство выбрать минимальное чистое решение.

## Task identity

Номер берётся из `ROADMAP.md` и не переиспользуется:

```text
Task #001 — Первая расчистка участка
```

- первая строка prompt, branch и PR title сохраняют эту идентичность;
- ветка: `task/001-first-debris-clear`;
- PR body начинается с `Task: Task #001 — Первая расчистка участка`;
- после открытия PR пользовательская ссылка имеет Task-first формат: `Task #001 — Первая расчистка участка (PR #81)`;
- repair сохраняет исходный Task number.

## Компактное ТЗ для Codex

Routine prompt содержит:

1. `Task #N — название`;
2. цель и наблюдаемый результат;
3. затронутую существующую архитектуру — только существенные точки;
4. требования и критические ограничения;
5. acceptance, которое нельзя уверенно вывести из кода;
6. границы scope, когда существует реальный риск расползания.

Ориентир для обычной комплексной задачи — 40–80 строк. Для strict-risk задачи допустимо 80–140 строк, если объём вызван migration, security, dependency, asset contract или несколькими зависимыми системами.

Не копировать в prompt:

- общие правила из `AGENTS.md`;
- полный список файлов, если Codex может найти их поиском;
- реализацию по функциям и классам без архитектурной необходимости;
- полный набор существующих проверок;
- PR template и Git lifecycle;
- exhaustive unit, E2E, manual runtime и screenshot matrix для одного риска.

## Выбор lane

Fast lane используется по умолчанию для gameplay, UI, content, config и bounded fixes.

Strict lane нужен для persistence/schema migration, central state ownership, broad movement/input/collision contract, dependencies/workflows/deployment/security, внешних assets/licensing или зависимых PR.

Размер prompt или diff сам по себе не делает задачу strict.

## Evidence

Один material risk получает одно сильное доказательство:

- точные значения и переходы — automated check;
- визуальное ощущение — focused runtime inspection;
- layout regression — один или два screenshot;
- полный repository contract — PR CI.

## Передача в Codex

После согласования результата Лид выдаёт одно самодостаточное ТЗ. Codex сам завершает branch → Ready PR → CI → repair при реальном failure → merge и сообщает итог пользователю.

Если пользователь прямо просит Лида выполнить отдельную операцию, Лид доводит собственную операцию до проверенного результата. Отдельный Integrator используется только по явной просьбе пользователя для независимой приёмки или координации нескольких зависимых PR.

## Пользовательские файлы

Для внешнего бинарника Лид следует `BINARY_IMPORT.md`: проверяет attachment, импортирует фактический runtime-файл и передаёт Codex точный путь. Остальные задачи не читают binary-import инфраструктуру.
