# NestledBurrow: roadmap и текущее исполнение

## Назначение

`ROADMAP.md` отвечает на два вопроса:

1. что мы делаем прямо сейчас;
2. что собираемся проверять следующим.

Состояние уже существующих механик и пользовательский verdict принадлежат `GAME.md`. Техническое доказательство реализации принадлежит коду, тестам и merged PR.

## Идентификаторы задач

Каждая согласованная работа получает постоянный номер до выдачи Codex prompt.

Канонический формат:

```text
Task #001 — Первая расчистка участка
```

- Номера назначаются последовательно и не переиспользуются.
- Один пользовательский результат сохраняет один Task number от prompt до merge и пользовательского verdict.
- Первая строка Codex prompt, branch, PR title, PR body и отчёты ролей сохраняют одну идентичность.
- Ветка: `task/<номер>-<slug>`.
- После открытия PR используется формат `PR #81 · Task #001 — Первая расчистка участка`.
- Base SHA не хранится здесь как долгоживущий факт: Лид проверяет актуальный `main` непосредственно перед запуском.

**Следующий свободный номер:** `Task #020`.

## Статусы

- `Запланировано` — направление согласовано, prompt ещё не выдан.
- `Готово к запуску` — самодостаточный prompt подготовлен.
- `В работе` — Codex выполняет задачу.
- `PR открыт` — результат доступен для приёмки.
- `Смёржено, ждёт теста` — код в `main`, пользователь ещё не дал verdict.
- `Принято` — пользователь подтвердил направление.
- `Переделать` — направление сохраняется, текущий результат не подходит.
- `Отклонено` — направление прекращено.
- `Завершено` — техническая или процессная работа завершена.
- `Заблокировано` — указан конкретный blocker.

## Сейчас

### Task #019 — Пересобрать игровой HUD

**Статус:** `Готово к запуску`

**Продуктовый вопрос:** перестаёт ли HUD конкурировать сам с собой и остаётся ли каждая постоянная функция мгновенно читаемой в нативном viewport `320×180`?

**Наблюдаемый результат:**

- в левом верхнем углу остаётся одна кнопка `Опции` / `Options`; отдельные постоянные кнопки `Новая игра`, sound и language из верхней полосы удаляются;
- панель опций содержит существующие настройки master/music/effects, переключение RU/EN и запуск существующего confirm-flow новой игры;
- локализованные часы находятся отдельно по центру верхнего края;
- fullscreen остаётся отдельной компактной кнопкой в правом верхнем углу;
- справа по центру находится неинтерактивная компактная панель текущих ресурсов с каноническими визуалами бревна и рубина и числовыми значениями;
- справа от панели ресурсов находится отдельная вертикальная шкала энергии, заполняемая снизу вверх;
- при успешной interaction-трате, оставившей меньше `15%` энергии, вертикально трясётся вся шкала энергии целиком, а не текст;
- interaction prompt сохраняет текущий формат и располагается в правом нижнем углу;
- build label не занимает постоянное место в игровом HUD;
- normal, options, confirmation и dialogue states не создают наложений, обрезанного текста или конкурирующих hit areas.

**Строки `GAME.md`:**

- `Суперприветливый UX при множестве систем`;
- `RU/EN, HUD, audio и browser build`;
- `Энергия игрока`;
- `Ресурсы и инвентарь`.

**Граница:** переразложить существующие HUD-функции без добавления настоящего инвентаря, новых ресурсов, новых настроек, новых gameplay-механик или внешних assets.

## Обновление

- Лид назначает следующий свободный Task number, обновляет раздел `Сейчас` и только затем выдаёт prompt.
- Codex сохраняет Task number и название в branch, PR title и PR body.
- Интегратор связывает PR и задачу, после merge обновляет `GAME.md` и статус здесь.
- После пользовательского теста Лид фиксирует verdict и конкретный вывод.
- Завершённая работа не остаётся бесконечно висеть активной.
- GitHub issues не являются обязательной памятью проекта.
