# NestledBurrow: roadmap и текущее исполнение

## Назначение

`ROADMAP.md` отвечает на два вопроса:

1. что мы делаем прямо сейчас;
2. что собираемся проверять следующим.

Состояние уже существующих механик и пользовательский verdict принадлежат `GAME.md`. Техническое доказательство реализации принадлежит коду, тестам и merged PR.

## Идентификаторы задач

Каждая согласованная работа получает постоянный номер до выдачи Codex prompt.

Канонический формат:

```text
Task #001 — Первая расчистка участка
```

- Номера назначаются последовательно и не переиспользуются.
- Один пользовательский результат сохраняет один Task number от prompt до merge и пользовательского verdict.
- Первая строка Codex prompt, branch, PR title, PR body и отчёты ролей сохраняют одну идентичность.
- Ветка: `task/<номер>-<slug>`.
- После открытия PR используется формат `PR #81 · Task #001 — Первая расчистка участка`.
- Base SHA не хранится здесь как долгоживущий факт: Лид проверяет актуальный `main` непосредственно перед запуском.

**Следующий свободный номер:** `Task #019`.

## Статусы

- `Запланировано` — направление согласовано, prompt ещё не выдан.
- `Готово к запуску` — самодостаточный prompt подготовлен.
- `В работе` — Codex выполняет задачу.
- `PR открыт` — результат доступен для приёмки.
- `Смёржено, ждёт теста` — код в `main`, пользователь ещё не дал verdict.
- `Принято` — пользователь подтвердил направление.
- `Переделать` — направление сохраняется, текущий результат не подходит.
- `Отклонено` — направление прекращено.
- `Завершено` — техническая или процессная работа завершена.
- `Заблокировано` — указан конкретный blocker.

## Сейчас

### Task #018 — Усилить сумерки и усталость

**Статус:** `Готово к запуску`

**Продуктовый вопрос:** становятся ли вечер, ночь, утро и истощение выразительными и заранее читаемыми без резких визуальных или двигательных переключений?

**Наблюдаемый результат:**

- полноэкранный multiply-тон плавно проходит вечером через `#FFB380`, `#FF648B` и `#425CD4`, удерживает ночной цвет и с 04:00 до 09:00 проходит ту же последовательность обратно;
- цветовые переходы непрерывны на границах суток, применяются ко всему игровому viewport и сохраняют HUD читаемым поверх overlay;
- при энергии не ниже 25% игрок сохраняет 100% скорости, при энергии 1% и ниже — 25%; между этими точками целевая скорость уменьшается по нелинейной кривой с ускоряющимся падением возле нуля;
- фактическая скорость приходит к новому целевому значению через существующее ускорение/торможение, без мгновенного clamp текущей velocity;
- после каждого успешного interaction-удара, который действительно потратил энергию и оставил меньше 15% запаса, трясётся только текст энергии по вертикали;
- feedback срабатывает для брёвен и рубиновых жил, но не от пассивного расхода, сна, debug-изменений или неуспешного interaction.

**Строки `GAME.md`:**

- `Спокойный непрерывный ритм без дневной отсечки`;
- `Суперприветливый UX при множестве систем`;
- `Перемещение, камера, keyboard/mobile input`;
- `Энергия игрока`;
- `Непрерывное время, день/ночь и ускоренный сон`.

**Граница:** доработать только существующие day/night overlay, энергетическую скорость игрока и HUD-feedback. Не добавлять погоду, календарь, источники света, постпроцессинг, новые ресурсы, новые interaction-типы или изменения экономики.

### Task #015 — Оживить сон, часы и суточный цикл

**Статус:** `Смёржено, ждёт теста`

`PR #106 · Task #015 — Оживить сон, часы и суточный цикл` смёржен в `main` коммитом `7bd4eb989ce49f5730ea5e858acb1287d272090d`.

**Канонический технический контракт:** `tasks/015-live-sleep-clock-cycle.md`.

**Продуктовый вопрос:** ощущается ли сон как реальное ускорение живого мира, а не как бегущие цифры, и помогают ли плавный световой цикл и аккуратные action prompts интуитивно читать состояние мира?

**Наблюдаемый результат:**

- после автоматического или ручного пробуждения игрок оказывается на свободной точке рядом с кроватью и сразу может двигаться;
- время отображается как часы без подписи: `HH:mm` для RU и `h:mm AM/PM` для EN;
- обычный темп равен `80` игровым секундам за одну реальную секунду: полный игровой день занимает ровно `18` реальных минут, один игровой час — `45` реальных секунд;
- default awake drain согласован с этим темпом: `1` энергия каждые `7.2` реальной секунды, чтобы полный запас `100` энергии по-прежнему длился примерно `16` игровых часов;
- сон сохраняет ускорение x8, но теперь ускоряет NPC и остальную игровую симуляцию, а не только счётчик;
- с 18:00 до 21:00 мир плавно темнеет, с 05:00 до 08:00 плавно светлеет, ночью остаётся читаемым;
- освещение реализовано плавным светло-синим multiply-тоном игрового мира без окрашивания HUD;
- на карте есть две постоянные рубиновые жилы, добываемые по тому же пятиударному и энергетическому правилу, что бревна;
- каждая жила выдаёт один рубин на последнем ударе, сохраняет remaining hits/cleared, переживает reload и восстанавливается через `NEW GAME`;
- HUD показывает энергию, древесину, рубины и локализованные часы без секундной подписи и без `x8`;
- desktop interaction prompt использует единый формат `SPACE · Действие`, где `SPACE` всегда прописными, а действие — sentence case;
- бревно показывает `SPACE · Рубить` / `SPACE · Chop`, touch-версия — только `Рубить` / `Chop`;
- кровать и рубины используют тот же шаблон: `SPACE · Спать`, `SPACE · Добыть`, `SPACE · Sleep`, `SPACE · Mine`.

**Строки `GAME.md`:**

- `Спокойный непрерывный ритм без дневной отсечки`;
- `Суперприветливый UX при множестве систем`;
- `NPC, диалоги и мини-квест`;
- `Save/reload и NEW GAME`;
- `RU/EN, HUD, audio и browser build`;
- `Developer/runtime tuning`;
- `Энергия игрока`;
- `Ресурсы и инвентарь`;
- `Руда и редкие ресурсы`;
- `Непрерывное время, день/ночь и ускоренный сон`.

**Граница:** это один законченный sleep/time/resource/interaction-UX slice. Не добавлять календарь, сезоны, погоду, освещение от отдельных ламп, полноценную рудную систему, инструменты, инвентарь или процедурное размещение ресурсов.
