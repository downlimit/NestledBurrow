# NestledBurrow: roadmap и текущее исполнение

## Назначение

`ROADMAP.md` отвечает на два простых вопроса:

1. что мы делаем прямо сейчас;
2. что собираемся проверять следующим.

Состояние и качество уже существующих механик принадлежат `GAME.md`. Техническое доказательство реализации принадлежит коду, тестам и merged PR. Здесь не должно быть случайных issue-номеров, скрытых очередей или контекста, существующего только в чате.

## Идентификаторы задач

Каждая согласованная работа получает постоянный номер до выдачи Codex prompt.

Канонический формат:

```text
Task #001 — Первая расчистка участка
```

- Номера назначаются последовательно и никогда не переиспользуются.
- Номер не меняется при repair, rebase, новом commit или переходе задачи между ролями.
- Один пользовательский результат сохраняет один Task number от prompt до merge и пользовательского verdict.
- Новая задача получает новый номер только когда меняется сам пользовательский результат или scope становится самостоятельной работой.
- Номер PR является GitHub-адресом и не заменяет номер задачи.
- Любое совместное упоминание пишется так: `PR #81 · Task #001 — Первая расчистка участка`.
- Первая строка нового Codex prompt обязана дословно повторять `Task #<номер> — <название>`, чтобы автоматически создаваемый заголовок задачи Codex был узнаваемым и стандартизированным.
- Ветка: `task/<номер>-<slug>`, например `task/001-first-debris-clear`.
- Заголовок PR дословно совпадает с заголовком задачи: `Task #001 — Первая расчистка участка`.

**Следующий свободный номер:** `Task #008`.

## Статусы

- `Запланировано` — направление согласовано, но задача Codex ещё не выдана.
- `Готово к запуску` — самодостаточный prompt подготовлен.
- `В работе` — Codex выполняет задачу.
- `PR открыт` — результат доступен для приёмки, но ещё не находится в `main`.
- `Смёржено, ждёт теста` — код в `main`, пользователь ещё не оценил игровой результат.
- `Принято` — пользователь подтвердил игровое направление.
- `Переделать` — направление сохраняется, но текущий результат не подходит.
- `Отклонено` — направление прекращено.
- `Завершено` — техническая или процессная работа находится в `main` и не требует отдельного продуктового теста.
- `Заблокировано` — указан конкретный технический или внешний блокер.

## Сейчас

### Task #001 — Первая расчистка участка

**Статус:** `Смёржено, ждёт теста`

**Реализация:** `PR #86 · Task #001 — Первая расчистка участка` смёржен в `main` коммитом `87c7be151068dfb1144fdaaded3c35a8f817661d`. Старый дублирующий PR #81 закрыт без merge.

**Продуктовый вопрос:** приятно ли разбирать заброшенное пространство и возникает ли чувство, что игрок начинает присваивать остров?

**Что теперь работает:**

- в мире есть один читаемый упавший лог;
- игрок подходит, поворачивается и видит `Разобрать / Clear`;
- успешное действие расходует энергию, выдаёт древесину, даёт короткий feedback и удаляет объект;
- результат переживает save/reload;
- `NEW GAME` восстанавливает объект и стартовые значения, не сбрасывая язык и звук;
- при нехватке энергии нет частичной мутации;
- максимум энергии, стоимость, награда и refill доступны в runtime developer panel;
- desktop/mobile E2E и полный PR check-chain зелёные.

**Граница:** это один законченный playable slice, а не универсальная система ресурсов, инструментов, действий или инвентаря. Естественного восстановления энергии, нескольких типов мусора и полноценного участка ещё нет.

**Нужен пользовательский тест:** ощущение действия, читаемость лога и HUD, качество feedback, понятность энергии и древесины. После теста статус меняется на `Принято`, `Переделать` или `Отклонено`.

**Статус:** `Завершено`

`PR #85 · Task #006 — Стандартизировать идентичность задач` смёржен в `main` коммитом `c93f97c4561910453173d503b77890d947567f8c`.

Единый Task number теперь проходит через первую строку Codex prompt, branch, PR title/body, repair contract и отчёты ролей. Контракт защищён автоматической проверкой `check-task-identity-contract.mjs` внутри `npm run check:docs`.

### Task #007 — Автоматизировать review и merge gates

**Статус:** `PR открыт`

Fast lane должен автоматически выполнить squash merge только после успешных `Validate` и `Browser E2E`, актуального Codex Review для final head, отсутствия незакрытых Codex threads, корректной Task identity и явного отсутствия `strict-risk`. Strict lane никогда не сливается этим workflow и остаётся за Интегратором.

**Primary risk:** workflow получает право записи в `main`, поэтому политика проверяется pure-code тестом, исполняется только из доверенного `main`, не checkout-ит PR head и отклоняет fork PR.

## Обновление

- Лид назначает следующий свободный Task number, обновляет раздел `Сейчас` и только затем выдаёт новый Codex prompt.
- Codex сохраняет точные Task number и название в branch, PR title и PR body.
- Интегратор во всех сообщениях, repair contracts и merge-отчётах связывает PR и задачу форматом `PR #<номер> · Task #<номер> — <название>`.
- Интегратор после merge меняет статус, добавляет PR/merge evidence и обновляет фактическую зрелость в `GAME.md`.
- После пользовательского теста Лид фиксирует итог и следующий продуктовый вывод одновременно в `GAME.md` и здесь.
- Завершённая работа не остаётся бесконечно висеть активной.
- GitHub issues могут использоваться для обычных багов, но не являются обязательной памятью проекта и не участвуют в восстановлении нового Лида.
