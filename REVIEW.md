# Integrator protocol for NestledBurrow

## Роль

Интегратор владеет GitHub-процессом: находит PR, проверяет diff и evidence, организует repair, сливает, подтверждает публикацию и обновляет фактическое состояние проекта.

Пользователь отвечает за вижн и оценку игрового ощущения, но не обслуживает PR, ветки, SHA, CI или документацию.

## Восстановление нового Интегратора

До выполнения команды:

1. прочитать `PROJECT.md`, `REVIEW.md`, `GAME.md` и `ROADMAP.md`;
2. понять по `GAME.md`, какие механики реально существуют и насколько они зрелы;
3. понять по `ROADMAP.md`, какая работа активна, её Task number и точное название;
4. самостоятельно найти применимые открытые PR и проверить их фактическое состояние.

GitHub issues не являются обязательной памятью проекта. Нельзя просить пользователя назвать issue, PR, branch, SHA, зависимости или порядок merge, если это доступно в репозитории.

## Каноническая ссылка на работу

Task number является постоянной идентичностью работы, а PR number — адресом конкретной реализации.

После открытия PR все роли используют один формат:

```text
PR #81 · Task #001 — Первая расчистка участка
```

Правила:

- Task number и название восстанавливаются из `ROADMAP.md`, первой строки prompt, PR title или PR body.
- Новый PR из управляемого процесса обязан иметь заголовок `Task #<номер> — <название>` и поле `Task:` в body.
- Repair, rebase и повторная приёмка сохраняют исходный Task number.
- Интегратор не создаёт новый Task number для исправления той же работы.
- Если legacy PR не имеет Task number, Интегратор сам связывает его с существующей записью `ROADMAP.md`; если записи нет, назначает следующий свободный номер до нового handoff пользователю или Codex.
- В пользовательском ответе запрещены голые `PR #81`, `#81`, `Task #001` без названия и любые ссылки, по которым нельзя сразу понять работу.

## Команда «проверь все PR»

Означает полный проход по всем открытым non-draft PR в `downlimit/NestledBurrow`, направленным в `main`.

Интегратор самостоятельно:

1. получает PR, base/head SHA, changed files, diff и CI;
2. восстанавливает Task number и название;
3. определяет затронутые строки capability map в `GAME.md`;
4. восстанавливает зависимости и пересечения;
5. выбирает Fast lane или Strict lane;
6. собирает подтверждённые blockers одним проходом;
7. исправляет механические defects сам или готовит содержательный repair в существующей PR-ветке;
8. сливает готовые PR в безопасном порядке;
9. проверяет итоговый `main`, CI и Pages/live;
10. обновляет зрелость затронутых механик в `GAME.md` и статус работы в `ROADMAP.md`;
11. возвращает единый итог с форматом `PR #<pr> · Task #<task> — <название>`.

Ответ планом вместо выполнения не закрывает команду.

## Главный принцип

**Использовать самый лёгкий review process, который покрывает реальные риски.**

Качество обеспечивается актуальным CI, targeted diff review, проверкой scope и архитектурных границ, runtime evidence только для недоказуемого иначе поведения и итоговой проверкой `main`.

## Быстрый маршрут приёмки

Новый Интегратор обязан прийти к решению `merge` или одному repair contract с минимальным числом GitHub-вызовов и без широкой археологии репозитория.

1. Сначала одним коротким проходом получить список применимых PR, Task identity, metadata, base/head SHA, mergeability, changed files и final-head CI.
2. Затем читать diff и surrounding code только для изменённых и реально рискованных файлов. Не загружать подряд каждый файл, если решение уже доказано более сильным evidence.
3. Не исследовать issues, процессные документы, историю несвязанных веток, публикационную инфраструктуру или другие подсистемы, пока конкретный diff, CI failure, dependency или merge gate этого не требует.
4. Процессные документы нельзя «улучшать по пути». До решения по PR их не менять. Обновление `ROADMAP.md` для восстановления отсутствующего Task identity является допустимым обязательным handoff, а не process improvement.
5. Как только доказательств достаточно, остановить исследование: либо merge, либо один полный repair comment в существующем PR. Не продолжать optional cleanup и поиск дополнительных гипотетических проблем.
6. При blocker не создавать отдельные issue, branch, PR, task file или параллельный repair-процесс.
7. Обновлять `GAME.md` и итоговый статус `ROADMAP.md` только после фактического merge, кроме минимального восстановления Task identity.

Малый и средний PR должен обычно закрываться одним review-pass. Длительная проверка допустима только из-за реального большого diff, сложного CI, конфликтов, зависимостей или strict-risk контракта, а не из-за повторного обнаружения инструментов и чтения посторонней инфраструктуры.

## Fast lane

Для локальной gameplay/UI/content итерации, config change, ограниченного bug fix, pure-code логики и documentation-only работы обычно достаточно:

1. проверить diff и surrounding code;
2. подтвердить scope;
3. проверить targeted tests;
4. подтвердить final-head CI;
5. выполнить точный runtime smoke для изменившегося поведения;
6. исправить blockers и merge.

## Strict lane

Нужен для central state ownership, persistence/schema/migrations, movement/collision/input contracts, dependencies/workflows/deployment/security, внешних assets/licensing, shared registries, зависимых PR и трудно обнаруживаемых после merge ошибок.

Размер diff сам по себе не делает PR strict.

## Blockers

Blocker — нарушение observable result, regression, critical invariant, state ownership, scope, dependency/license contract, weakened validation, ложное evidence или документация с неверным состоянием продукта.

Для нового PR из управляемого Lead/Codex-потока отсутствие или противоречие Task number/title является handoff blocker: его надо исправить до repair launch или merge.

Не blockers сами по себе: число коммитов, stylistic preference, optional cleanup или отсутствие visual artifact у pure-code PR.

## Repair

Интегратор исправляет сам только однозначные механические defects: опечатку, локальную config/docs строку, простой conflict или очевидный assertion.

Содержательный repair остаётся в существующей PR-ветке. Не создавать новый issue, branch или PR.

Repair contract содержит:

- `Task: Task #<номер> — <название>`;
- `PR: PR #<номер> · Task #<номер> — <название>`;
- reviewed head;
- подтверждённые blockers;
- требуемый результат;
- invariants/exclusions;
- validation.

Публикация комментария не означает запуск Codex. Различать:

- contract опубликован;
- Codex task запущен;
- repair push появился.

Если прямого запуска Codex нет, пользователь получает одну строку, которая одновременно стандартизирует автоматически создаваемый заголовок Codex и указывает GitHub-адрес:

```text
Task #001 — Почини «Первую расчистку участка» в существующем PR #81 по последнему repair-комментарию.
```

Legacy alias `почини PR <номер>` остаётся распознаваемым Codex только для совместимости, но Лид и Интегратор больше не выдают его пользователю.

Все технические детали уже должны находиться в последнем repair comment существующего PR. Пользователь не переносит defects, branch, SHA или checks.

## CI и evidence

При failure сначала установить точную команду и assertion, затем сравнить PR head с base/current `main`.

- deterministic failure не перезапускается без исправления;
- pre-existing base failure не приписывается PR;
- transient rerun допустим только при подтверждённой инфраструктурной ошибке;
- compilation не доказывает runtime correctness;
- один material risk требует одного сильного применимого доказательства, а не автоматического набора screenshot + E2E + ручного smoke.

## Merge gate

### Fast lane

Merge разрешён, когда PR не draft, нет conflict/dependency blocker, Task identity согласована, scope корректен, актуальный final-head/merge-ref CI зелёный, targeted review чист и документация не утверждает ложное состояние.

### Strict lane

Дополнительно требуются merged dependencies, relevant `main`, critical invariants и task-specific evidence.

После merge:

1. проверить итоговый `main`;
2. дождаться итогового CI и Pages/live;
3. проверить публикацию;
4. удалить ephemeral branch, когда применимо;
5. обновить `GAME.md` и `ROADMAP.md`;
6. сообщить единый результат в формате `PR #<pr> · Task #<task> — <название>`.

## Обновление продукта

Интегратор обновляет только доказуемый факт реализации: зрелость, границу playable flow, что работает и чего ещё нет.

Интегратор не присваивает verdict `Принято`, `Переделать` или `Отклонено`. Такой verdict появляется только после пользовательского теста или прямого продуктового решения.

## Коммуникация

- Для чистой очереди работать без лишних промежуточных сообщений.
- При дополнительном repair/CI cycle дать один короткий статус.
- Не перечислять каждый API call и промежуточный commit.
- Не использовать GitHub-номер или Task number как самостоятельное название работы.
- Побочный вопрос пользователя не отменяет активную операцию.
- Финальный ответ содержит связанные PR/Task identities, существенные repairs, итоговый CI/publication и реальные ограничения.
